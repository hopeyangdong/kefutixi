前言：你和 AI 的关系正在变

🤔 先问一个问题

你现在是怎么用 AI 的？

大多数人的答案是：打开 ChatGPT / Claude / 豆包 → 打字提问 → 看回答 → 关掉。

这相当于你买了一辆特斯拉，但只用它听收音机。

🚗
汽车的演进类比：
手摇启动(Prompt) → 加油站+GPS(上下文工程) → 自动挡+倒车雷达(Agent) → 定速巡航(Skill) → L2 辅助驾驶(Harness) → L4 自动驾驶(Loop)

这门课会带你走完 六个台阶，每一层都建立在前一层之上。不跳级，不炫技，一层一层搭。

📐 六阶段全景图

📌 核心规律：每升一层，人类的角色就发生一次迁移 —— 从亲手干活的操作员，逐步退到只看结果的决策者。AI 承担的范围越来越大，但你给它的约束和标准也必须越来越精确。
01

Prompt 工程 2022末—2024上

💡 一句话理解

Prompt 工程 = 你说一句，AI 答一句。这是所有人接触大模型的第一步。你的整个输入（提示词）就是 AI 唯一的「世界」。

📱
类比：发短信。
你发一条短信给一个很聪明的朋友。他只看到这条短信，不知道你之前跟别人聊了什么，不知道你手上有什么资料。他的回答质量，100% 取决于你这条短信写得有多清楚。

🔑 核心技巧

技巧原理示例
角色设定 给 AI 一个身份，激活对应的知识 「你是一个有 10 年经验的客服培训师…」
明确格式 减少 AI 猜你要什么输出形式 「用表格对比 A 和 B，列出优劣」
给示例 Few-Shot，用例子定义标准 「好的回复像这样：…，差的像这样：…」
拆分步骤 避免一口吃成胖子 「第一步分析原因，第二步给方案，第三步写话术」

⚠️ 天花板在哪？

  • 无记忆：每次对话都是全新开始，昨天教它的今天全忘
  • 无数据：不知道你公司的文档、流程、规则
  • 无行动：只能回答文字，不能操作任何系统
  • 输入长度有限：早期 GPT-3.5 只有 4K token ≈ 3000 字
🎯 天花板公式：Prompt 的上限 = 你描述能力的上限 × 模型知识的上限。当你发现「怎么写都差点意思」的时候，就该往第二层走了。

🧪 小测验

你想让 AI 帮你分析一份 100 页的客服手册，Prompt 工程能搞定吗？

A. 能，把 100 页全粘贴进去就行
B. 能，分 10 次粘贴，每次 10 页
C. 不太行，需要上下文工程（RAG）来解决
✅ 正确！100 页超出了 token 限制，分批又会丢失上下文。这正是上下文工程要解决的问题 —— 让 AI 动态检索你需要的段落，而不是全塞进去。
02

上下文工程 2023—2024

💡 一句话理解

上下文工程 = 在 AI 回答之前，先帮它把「教材」准备好。

Prompt 解决的是「怎么问」，上下文工程解决的是「拿什么来答」。它包含两大核心技术：

🧠
CoT 思维链

教 AI 怎么想
「先分析原因→再给方案→最后总结」

🔍
RAG 检索增强

给 AI 看什么
从知识库里捞出相关段落塞进去

📖 RAG 是什么？给小白画张图

💡 一句话总结 RAG：不是把 100 页全塞给 AI，而是只给它最相关的几段。就像你不会把整个图书馆搬给朋友，而是帮他找好那几页。

🆚 Prompt vs 上下文工程

对比维度Prompt 工程上下文工程
知识来源AI 自己的训练数据你的文档、数据库
能处理多大材料几千字无上限（检索后只取片段）
回答是否可溯源不能，可能瞎编可以，能标注出处
实时性截止到训练日期知识库实时更新
部署难度零（直接聊天）中等（需建索引）

🎯 适用场景举例

  • 客服知识库问答（100+ 页客服手册 → RAG 检索 → AI 精准回答）
  • 企业内部搜索（在 10 万篇学城文档中找答案）
  • 代码库理解（给 AI 喂相关代码片段，而不是整个仓库）
  • 复杂推理（CoT：先拆解问题 → 逐步推导 → 综合结论）
03

Agent 智能体 2023中—2024

💡 一句话理解

Agent = AI 不只回答问题，还能动手干活了。

🏨
类比：从前台客服到管家。
Prompt 阶段，AI 是酒店前台 —— 你问什么答什么。Agent 阶段，AI 变成管家 —— 你说「帮我订明天的机票」，他会自己去查航班、比价、下单，而不是回你一句「您可以打这个电话」。

🔧 关键技术：Function Calling

大模型本身只会「说话」。但通过 Function Calling（函数调用），AI 可以指挥外部工具：

🔍
搜索引擎

去网上查最新信息

💻
写代码并运行

写 Python 脚本分析数据

📊
查数据库

执行 SQL 查询看板数据

📤
发消息

把结果发到群聊/邮件

// AI 的内心独白（简化示意）

// 用户说："查一下昨天解决率多少"
思考: 用户要查解决率，我需要调用 BI 查询工具
调用: query_bi("SELECT solve_rate FROM dashboard WHERE date = yesterday")
结果: 87.3%
回答: 昨天解决率 87.3%，环比上升 0.5pp ✅

⚠️ Agent 阶段的痛点

Agent 能干活了，但有三个明显的毛病：

  • ❌ 不可复用：你教了它一次怎么查解决率，下次换个人问，还得重新教
  • ❌ 不够稳定：同一个任务，跑 10 次可能有 3 次走弯路
  • ❌ 无法自检：做错了不知道错了，把错误结果自信满满地交给你
🎯 问题本质：Agent 像一个很聪明但没有经验的实习生 —— 什么都能干一点，但缺少标准化的操作手册（Skill）和质量控制体系（Harness）。
04

Skill 技能封装 2024下半年

💡 一句话理解

Skill = 把成功经验固化成可复用的「操作手册」。

📖
类比：从「口口相传」到「SOP 手册」。
Agent 阶段靠个人悟性，Skill 阶段把最佳实践写成标准文档。新人来了翻手册就行，不用老师傅一对一带。

🧩 Skill 的本质结构

一个 Skill 通常包含这四部分：

🎯
触发条件

什么时候该用这个技能

📝
操作步骤

1-2-3 怎么做

⚠️
避坑指南

常见错误和陷阱

✅
验收标准

怎么判断做对了

# 示例：一个客服解决率日报 Skill

触发条件: 每天早上 9:00 自动触发
步骤 1: 调用 BI 接口查昨天 vs 前天解决率
步骤 2: 计算各 L1 场景的变化贡献
步骤 3: 找出跌幅 Top3 场景并下钻到 L4
步骤 4: 生成桥图 + 文字摘要
步骤 5: 推送到大象群

避坑: 周一不跟周日比，要跟上周一比
验收: 数字与看板一致，偏差 < 0.1pp

🆚 Agent vs Skill

对比维度裸 AgentAgent + Skill
做同一件事每次临场发挥按手册执行
稳定性60-70%（看状态）90%+
可复用换人要重新教任何人直接用
可迭代经验在人脑里Skill 越用越准
知识传承❌ 老师傅走了就没了✅ 写进 Skill 永远在
💡 关键洞察：Skill 不是「限制」AI，而是让 AI 站在巨人肩膀上。你的最佳实践被打包，AI 每次都从 90 分起步，而不是从 0 开始摸索。
05

Harness 运行外骨骼 2024末—2025

💡 一句话理解

Harness = 给 AI 穿上一套铠甲——不是增强能力，而是防止翻车。

🏗️
类比：建筑工地的安全体系。
工人（Agent）有了施工手册（Skill），但工地还需要安全帽、防护网、监理、验收流程。Harness 就是这套安全体系 —— 让 AI 在「可控范围内」自由发挥。

🛡️ Harness 的四大组件

🎯 为什么 Harness 不可跳过？

很多团队犯的错是：有了 Agent + Skill 就直接上 Loop。结果：

  • 🔥 AI 半夜跑出 bug，没人发现，错误数据发了一整天
  • 🔥 AI 执行了不该执行的操作（没有护栏）
  • 🔥 一个环节失败，整条链路崩溃（没有容错）
  • 🔥 出了问题不知道哪一步错了（没有监控）
⚡ 一句话：没有 Harness 的 Loop = 一辆没有刹车的自动驾驶汽车。速度越快，翻车越惨。

🧪 小测验

你部署了一个 AI 每天自动生成解决率日报。某天数据源异常返回 0，AI 把「解决率 0%」推到了群里。这属于缺少哪个 Harness 组件？

A. 护栏 Guardrail
B. 质检 Validator
C. 容错 Retry
D. 监控 Monitor
✅ 正确！应该有一个 Validator：「如果解决率突然变化超过 ±10pp，暂停推送，标记人工复核」。护栏管的是「能不能做」，质检管的是「做对没有」。
06

Loop 自主循环 2025末至今

💡 一句话理解

Loop = AI 不再等你指令，而是自己跑起来、自己判断、自己交付，你只管看结果。

🏭
类比：从手工作坊到自动化工厂。
前五层是在「教一个人怎么干活」。Loop 是把整条生产线自动化 —— 原料进去、产品出来，你只在异常告警时才介入。
💡 核心转变：从「你盯着 AI 干」→「AI 自己干完给你结果」

🔄 Claude 定义的四种 Loop 类型

这四种不是「选一个」，而是自动化程度从低到高的四个档位：

01 回合制循环 — 半自动

你提一个需求，AI 自己迭代多轮直到搞定。

🔍
读代码
✏️
改代码
🔄
跑测试
✅
检查结果

🔑 关键机制：写 SKILL.md 设验收标准，Claude 自己验证自己

你的角色：提需求 → 等结果 → 验收。中间过程不用管。

02 目标循环 — 准自动

设定量化目标，AI 自动评估是否达标，没达标就继续优化。

/goal 测试覆盖率 ≥ 80%

// AI 内心：当前 62% → 写测试 → 71% → 继续 → 83% ✅ 达标！

🔑 关键机制：有标准、有评估，自动收敛

你的角色：只定义「成功是什么样」，不管「怎么到达」。

03 定时循环 — 自动

设定时间表，AI 定时自动执行任务，无需任何触发。

/schedule 每天 09:00 总结 Slack 消息

// 每天早上你打开手机，AI 已经把昨天的重要消息整理好了

🔑 关键机制：/loop 或 /schedule，周期执行，省心省力

你的角色：设置一次，之后只看结果。

04 主动循环 — 全自动

AI 自己发现问题 → 提出多个方案 → 自动对比择优 → 交付最优解。

🔍
搜查 Bug
📊
分类分析
🔄
自动修复
✅
回归验证

🔑 关键机制：方案 A / B / C 并行探索 + AI 裁判 → 最优方案自动交付 🏆

你的角色：只管看最终结果。AI 是选手，也是裁判。

🆚 四种 Loop 对比一览

Loop 类型自动化程度人类参与适用场景
① 回合制 半自动 🟡 提需求 + 验收 写代码、写文档、修 Bug
② 目标循环 准自动 🟢 定目标 性能优化、指标达标
③ 定时循环 自动 🔵 设置一次 日报、监控、定期巡检
④ 主动循环 全自动 🟣 只看结果 自动修复、方案择优
⚠️ 重要提醒：重点不是循环越多越好！
Loop 不是目的，是手段。选哪种 Loop 取决于你的场景需要多少自动化。一个精准的回合制循环，可能比一个失控的全自动循环好 10 倍。

🧪 小测验

你想让 AI 每天早上自动分析昨天的客服解决率，生成日报推送到群里。应该用哪种 Loop？

A. 回合制循环 —— 每天手动跟 AI 说「帮我看看」
B. 目标循环 —— 设定「解决率 ≥ 85%」为目标
C. 定时循环 —— 设置 /schedule 每天 09:00 自动执行
D. 主动循环 —— 让 AI 自己决定什么时候生成
✅ 正确！这是一个固定时间 + 固定流程的任务，定时循环最合适。目标循环适合「优化到某个值」的场景，而日报是「按时交付」的场景。

课程总结：六层台阶 + 一个心法

📊 六阶段演进对照表

阶段核心能力你的角色AI 能做AI 不能做
① Prompt 表达清楚需求 操作员 回答问题 记忆、查资料、动手
② 上下文工程 给 AI 配知识 策展人 基于你的数据回答 执行操作、自我检查
③ Agent 调用工具干活 指挥官 查数据、写代码、发消息 稳定复用、自我质检
④ Skill 固化最佳实践 架构师 按标准流程执行 异常恢复、自动决策
⑤ Harness 质控 + 容错 督察 自检、重试、告警 无人值守运行
⑥ Loop 自主闭环 决策者 自动运行 + 自动交付 （取决于前五层的建设）

🧠 一个心法

「控制权上移，标准下沉」

每升一层，你交出去的是执行权（AI 替你干）
每升一层，你必须写下更精确的标准（AI 按你的规矩干）

🎯 最终结论：Loop 的前提不是技术多牛，而是你在前五层——Prompt、上下文、Agent、Skill、Harness——每一层都建设扎实了。地基不牢，自动化程度越高，翻车越猛。

🗺️ 自测：你在哪一层？

你还在手动打字问 AI？

→ 你在第 1 层。先学好 Prompt 四要素：角色、格式、示例、步骤。

你开始给 AI 喂自己的文档了？

→ 你在第 2 层。试试用 RAG 让 AI 检索你的知识库。

你让 AI 查数据库、调接口了？

→ 你在第 3 层。把重复的任务固化成 Skill。

你写了标准操作手册给 AI？

→ 你在第 4 层。加上质检、护栏、监控，搭建 Harness。

你的 AI 有自检和告警了？

→ 你在第 5 层。可以开始跑 Loop 了！

你只看结果，AI 全自动闭环？

→ 恭喜，你到顶了。持续优化 Skill 和 Harness 即可 🏆
