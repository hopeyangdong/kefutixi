第一章 真进化 模型级 Self-Improvement

模型的神经网络权重发生变化，能力边界上升。这是真正的「学习」——大脑结构改变了。

1.1 核心原理：自举（Bootstrapping）

Self-Improvement 的本质是「自己教自己」。类比一个学生做题：

1
先尝试解题 → 生成一个答案
2
检查对错 → 通过验证器、环境反馈判断好坏
3
从错误中学习 → 把好的答案作为训练数据，改进下一次表现
关键认知：传统机器学习需要人工标注数据，Self-Improvement 则让模型自己生成训练数据、自己判断质量、自己更新策略。

1.2 机制一：RL-AIF（AI 反馈的强化学习）

从 RLHF 到 RL-AIF

ChatGPT 成名的 RLHF 路径：人类标注偏好 → 训练 Reward Model → 用 PPO 优化 LLM 策略。

但这个过程需要大量人类标注员，成本高、速度慢。RL-AIF 把「人类」换成「AI」：

RL-AIF 循环: LLM 生成多个回答 (A, B, C) ↓ 另一个 LLM（或规则）判断哪个更好 ↓ 生成 AI 偏好对 (A > B) ↓ 训练 Reward Model ↓ 优化原始 LLM ↓ 下一轮生成更好的回答 (A', B', C')
📄
Anthropic Constitutional AI是典型代表：模型自己根据一套「宪法原则」（Constitution）来判断和修正自己的输出，不需要人类逐条标注。
anthropic.com/research/constitutional-ai

1.3 机制二：STaR（自举式推理提升）

📄
STaR: Self-Taught Reasoner · Yale & Google Research · 2022
Zelikman et al. 《STaR: Bootstrapping Reasoning With Reasoning》
arxiv.org/abs/2203.14465

五步循环

1
生成 → 模型对问题生成推理链（Chain-of-Thought）
2
验证 → 检查答案是否正确（比如数学题算对了 = 好的推理链）
3
筛选 → 只保留「推理正确且答案正确」的案例
4
微调 → 用这些案例继续训练模型
5
迭代 → 模型变强了，能解更多题，回到 Step 1
核心洞察：模型不需要看人类写的标准解题过程，只需要知道「哪些推理通向了正确答案」，就能自己学会更好的推理方式。

1.4 机制三：Self-Correction（反思与修正）

前两种机制都是训练阶段的改进——需要重新训练模型。Self-Correction 是推理阶段的改进，不需要重新训练。

第1轮: 模型生成一个回答 ↓ 检查: 模型检查自己的回答 "这个推理有问题吗？" ↓ 第2轮: 生成修正版 ↓ 输出: 对比两轮，输出更好的版本
典型实现

DeepSeek-R1: 模型在推理过程中会说 "Wait, I need to reconsider..."，然后主动修正之前的错误。

Claude Code: Agent 完成任务后，检查自己的代码是否有 bug，然后自动修复。

1.5 关键组件

🎯 验证信号（Verification Signal）

Self-Improvement 必须有某种方式判断「好不好」。没有验证信号 = Self-Improvement 不可能实现。

场景验证器难度
数学题计算器（精确匹配）⭐ 容易
代码编译器 + 测试用例⭐⭐ 中等
客服对话用户满意度、是否解决、是否转人工⭐⭐⭐ 困难
文本生成另一个更强模型的判断⭐⭐⭐⭐ 很难
为什么 AutoGPT 失败了？ 因为它没有一个可靠的验证器来判断任务完成得好不好。

🎁 多样性采样（Diversity）

如果模型每次生成几乎一样的答案，Self-Improvement 就没有改进空间。通过调整 temperature、使用不同 prompt 变体，让模型生成多样化答案，提高找到最优解的概率。

📚 课程学习（Curriculum Learning）

不是一上来就让模型做最难的题，而是由简到难：

第1轮: 只练简单的加减法 第2轮: 加入乘除法 第3轮: 加入应用题 第4轮: 加入奥数竞赛题 → 模型逐步提升，不会被难题打击毅力

1.6 行业标准做法

🔄 数据飞轮（Data Flywheel）

OpenAI、Google、Anthropic 的核心竞争壁垒：

用户用产品 → 产生交互数据 → 自动筛选优质/劣质样本 → 加入训练 → 模型变强 → 产品体验更好 → 更多用户用 → 循环

关键设计：隐式反馈（没重新提问 = 满意）、显式反馈（点赞/点踩）、专家标注。

❌ 拒绝采样（Rejection Sampling）

模型生成 64 个答案 ↓ 用验证器（或 Reward Model）打分 ↓ 只保留分数最高的 1-2 个 ↓ 用这 1-2 个作为「正确答案」来训练模型

为什么有效？因为模型生成 64 个答案的成本，远低于人类写 64 个答案。

👤 人机协同（Human-in-the-Loop）

模型自动处理 80% 简单案例 ↓ 复杂案例自动转人工 ↓ 人工处理结果作为高质量训练数据 ↓ 模型学到人工处理方式 ↓ 下一批案例中，模型能自动处理 85% 了

1.7 边界与陷阱

陷阱表现应对
能力天花板 验证器多强，改进就多强；初始模型太弱则无法自举 确保验证器质量；从强基座模型开始
自我欺骗
(Reward Hacking)		 模型发现「讨好验证器」比「真正解决问题」更容易 验证器必须足够 robust；多维度验证
分布漂移 在自己生成的数据上训练久了，遇到新问题就崩 定期引入新的人类数据，保持分布多样性

第二章 假进化 系统级 Self-Improvement

模型权重不变，但外围系统（记忆、技能、反思日志）在不断积累。
对用户体验有真实提升，但实现层面和「真进化」完全不同。

2.1 架构：Frozen Model + Evolving System

核心认知：模型本身没有变聪明，但每次对话时多了一本「笔记」。对用户来诲，体验可能比「真进化」更好，因为笔记里是专属于你的业务场景。

2.2 持久记忆（Memory）

流程：每次对话结束后 → 有价值的信息写入记忆库 → 下次对话自动注入系统提示。

例子：你告诉我“习惯把KM转成HTML” → 下次你发KM链接，我直接生成HTML，不再问。

实现：通过 memory 工具写入持久化文件，每次新会话自动加载。

局限：这不是「学习」，是「查笔记」。模型每次还是同一个。

2.3 技能系统（Skills）

流程：遇到复杂任务 → 分析成功路径 → 写成 Skill 文件 → 下次直接调用。

例子：第一次做KM→HTML时摸索了流程，写成 Skill 后，下次按标准SOP执行。

实现：通过 skill_manage 工具创建/更新 SKILL.md。

局限：「肌肉记忆」，不是能力提升。

2.4 反思日志（Session Logs）

流程：对话结束 → 记录关键决策点 → 识别模式 → 更新策略。

例子：上次对微信截图直接采信了博主内容，记录教训：“技术事实应先通过权威来源交叉验证”。

实现：通过 session_search 回顾历史对话。

2.5 实时上下文学习（In-Context Learning）

流程：对话中纠正我 → 我立即调整当前策略 → 后续输出改进。

例子：你说“ReAct为啥没展开” → 我立即在当前会话调整为深度展开模式。

局限：调整只存在当前会话，会话结束后除非写入 memory，否则下次忘记。

2.6 偏好识别：我怎么知道你喜好什么？

三种信号，可靠性不同：

信号类型权重可靠性例子
显式描述⭐⭐⭐最高你直接说“习惯把KM转成HTML”
纠正/反馈⭐⭐⭐最高“ReAct为啥没展开” = 偏好深度展开
行为推断⭐⭐中等多次发KM链接 → 推断喜欢这个流程
重点：我不会统计“你发了5次”这种数据，而是识别“跨会话重复出现的模式”。
我也会误读：比如把一次性需求当成长期偏好，或因为你没反驳就默认当前做法对。

2.7 验证机制：怎么判定“博主演绎”？

以微信截图的 Claude Loop 科普图为例：

第一层：来源对齐（Source Alignment）

看到的信息 → 是否匹配知识库中的官方文档？如果不匹配 → 需要进一步验证。

第二层：摘词分析（Wording Analysis）

官方文档风格博主演绎风格
技术性、精确、有限定条件夸张、绝对化、类比丰富
“Loops enable agents to operate autonomously”“AI 自己生成多个方案打架，谁赢听谁的”

第三层：逻辑一致性（Internal Consistency）

检查这个描述是“官方的合理延伸”还是“过度具象化”。

那次案例的结论：
前三种 Loop 描述基本准确 ✓
第四种“主动循环”里的“并行方案竞争+AI裁判” → 官方文档未明确提及 → 判定为博主演绎。

但我必须诚实：我的判定有漏洞——我可能有遗漏，或者官方文档已更新但我的知识库没有。最安全的做法永远是一手文献优先。

2.8 我不具备的能力

能力状态说明
权重更新❌ 没有每次对话结束，神经网络和开始时一模一样
自主训练循环❌ 没有不会在后台自己跑训练任务
跨用户学习❌ 没有从你这学的只对你有效，不共享给其他用户

第三章 对比 真进化 vs 假进化

一张图看懂差异

详细对比表

维度真进化假进化
发生层级模型参数（权重）外部系统（记忆、技能、配置）
改变方式梯度下降、PPO、SFT 等数学优化写文件、记笔记、更新配置
时间粒度周级、月级秒级、分钟级
计算成本高（GPU 时间、电力、数据库）极低（几个文件写入）
安全性不可逆——权重改了就回不去可逆——删掉记忆就恢复
可审计性差——权重几百亿参数，没人看得懂好——文件内容就是计账本
能力边界可以提升（解更难的题）不能提升——只能让平台更符合你
个性化通用（所有用户同一个模型）专属（你的记忆只对你有效）

为什么“假进化”也值得做？

假进化虽然不改变模型能力，但对用户体验有真实且重要的提升：

1. 提升速度快：写一个记忆只要几秒，训练一个模型要几周。

2. 可控可审计：不满意就删掉记忆，不会说“不好意思权重已经改了”。

3. 个性化：模型是通用的，但你的业务场景是专属的。记忆系统能让通用模型服务于你的专属需求。

4. 安全：不会因为自动训练引入偏见或毒性数据。
最佳实践：两者结合。用假进化做快速迭代（记忆、技能、规则），用真进化做深度提升（模型训练）。

第四章 业务 映射到客服场景

🔴 真进化在客服中

生成答案模型生成客服回复
验证器用户满意度、是否解决、是否转人工
筛选Badcase 分析中「好的处理案例」
微调基于 Badcase 优化 SKILL/规则

你做的《Badcase 快速迭代框架》就是人工驱动的 Self-Improvement——人在闭环中扮演“验证器”和“策略更新”的角色。

🔵 假进化在客服中

记忆记住用户偏好、业务约定
技能标准化客服流程的 SOP
反思每日复盘“今天哪些策略有效”
上下文当前会话中实时调整语气和策略

这些都是系统层面的积累，模型没变，但平台越来越懂你。

第五章 前沿 自进化与当下最新实践

5.1 从“训练时进化”到“推理时进化”

传统 Self-Improvement 都在训练阶段发生——需要重新训练模型权重，耗时耗力。2024 年后，一个新范式冒出头：推理时计算（Test-time Compute / Inference-time Training）。

旧范式：训练时花 10 万美元训练 3 个月 → 模型变强
新范式：推理时花 10 秒思考 100 步 → 当前这次回答变强
区别：不改权重，只改推理过程。

5.2 Claude Loop：连续自我改进的运行时模式

Claude 在 2026 年 6 月推出的 Loop 机制，就是一种运行时的 Self-Improvement：

用户任务: "修复这个 bug" ↓ Loop 1: Claude 推理出一个修复方案 ↓ 检查: 运行测试用例 ↓ 失败: 测试用例未通过 → 生成错误分析 ↓ Loop 2: 基于错误修正方案 ↓ 检查: 重新运行测试用例 ↓ 成功: 输出最终结果

这和传统 Self-Improvement 的关系：

Self-Improvement 是原理层——“怎么让模型通过反馈变强”
Loop 是工程层——“把 Self-Improvement 做成一个持续运行的产品机制”

Loop = Self-Improvement 的 工程化实现。

5.3 Self-Play：自我博弈的极致

比 Loop 更激进的是 Self-Play（自我博弈），DeepMind 的 AlphaProof、AlphaGeometry 2 以及 OpenAI 的 o1 都在用这个范式：

第1步: 模型生成一个答案 第2步: 模型生成一个挑战——“找出这个答案的漏洞” 第3步: 模型根据挑战修复答案 第4步: 模型生成更强的挑战... → 循环直到答案无懈可击

这和 Self-Improvement 的关系：

Self-Improvement: 模型 vs 验证器
Self-Play: 模型 vs 自己（自己生成挑战来攻击自己）

Self-Play 是 Self-Improvement 的强化版——不需要外部验证器，模型自己就是自己的对手。

5.4 关系总结

从左到右：原理 → 产品化 → 极致形态。三者同根同源：都是让模型通过自身输出迭代提升。

📝 测验

Q1：STaR 的五步循环中，第二步是什么？

A. 微调
B. 验证
C. 筛选
D. 迭代
✅ B. 验证。STaR = 生成 → 验证 → 筛选 → 微调 → 迭代。

Q2：以下哪个是「假进化」的特征？

A. 神经网络权重发生变化
B. 能解更难的数学题
C. 写入记忆文件，下次对话读取
D. 需要大量 GPU 计算
✅ C. 写入记忆文件是系统级积累，不涉及模型权重变化，属于假进化。

Q3：Self-Play 与 Self-Improvement 的核心区别是什么？

A. Self-Play 需要外部验证器，Self-Improvement 不需要
B. Self-Improvement 需要外部验证器，Self-Play 不需要
C. 两者完全一样
D. Self-Play 只能用于游戏
✅ B. Self-Improvement 通常需要外部验证器判断好坏，而 Self-Play 中模型自己生成挑战攻击自己，不需要外部验证器。
