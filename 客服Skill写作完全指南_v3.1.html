01
Skill 是什么
Skill 是给大模型用的推理脚手架——不是给人看的文档，也不是 SOP 或话术库。
1.1 与其他文档的根本区别
类型输入输出决策逻辑面向谁
FAQ用户问题固定答案没有用户
SOP情况描述操作步骤有，需人来判断人工客服
话术库场景标签推荐话术没有人工客服
Skill当前对话 + 系统信号判断 + 动作 + 回复有，模型自主执行大模型
1.2 完整生命周期
大多数团队只做了前两步就算完。没有监控和迭代，Skill 会随业务变化逐渐失效。
1
准备
收集数据理解业务
直接跳过写完返工
2
写作
按指南方法完成内容
只写内容不写约束
3
测试
历史对话验证
用例不够上线踩坑
4
上线
灰度部署真实用户
全量上线没灰度
5
监控
追踪升级率解决率
没指标凭感觉
6
迭代
根据数据改进
发现问题不知从哪改
1.4 Skill 与 System Prompt 的关系（方法论层）
System Prompt 是宪法（永远存在，定义身份/格式/全局准则），Skill 是具体法律（按需加载，定义业务规则）。工程实现细节见附录 B。
System PromptSkill
内容身份、输出格式、全局行为准则、动作规则特定场景的业务判断、处理流程、工具清单
生命周期永远存在，每次对话都加载按需注入，只在相关场景加载
变更频率极少变更（产品架构级）随业务规则迭代（运营级）
维护角色工程师 + 产品架构师运营 + 业务产品经理

三条必须遵守的原则

#原则违反后果
1Skill 只写本场景特有内容，不重复 System Prompt每次迭代要多处同步，遗漏时行为不一致
2业务规则按需加载，不堆进 System Prompt推理质量下降，成本线性上升，无法单独迭代
3Skill 间冲突优先级写在 System Prompt，不分散在各 Skill 内多 Skill 加载时模型无所适从
02
动笔前的准备工作
最常见的错误是直接开始写。正确顺序：先绘制用户旅程 → 从旅程推导义务矩阵 → 确定 Skill 覆盖的场景清单。
2.1 所有进线场景的顶层分类
先用这张框架穷举所有可能进线的情况——这是 Skill 体系 MECE 覆盖的起点。
订单相关（Skill 体系的核心覆盖范围）
售前
选购咨询·优惠
→
履约中
催单·超时·骑手异常
→
售后
少送·品质·退赔
非订单（通常给自助入口，不单建 Skill）
App 报错
发票开具
账号问题
会员权益
重心提示：即时零售业务 履约中 占 40–60%，售后 占 15–25%，售前极低。Skill 体系建设优先级应与进线分布对齐。
2.2 义务矩阵：从旅程推导责任归属

义务矩阵模板（阶段 × 角色）

阶段用户商家 / 供应方配送方平台
下单提供正确收货地址确保定位服务正常

每格填：必须做什么 → 违约的信号字段是什么 → 违约后谁负责。这张矩阵直接成为履约 Skill 的诊断骨架。
03
先设计体系，再写单个 Skill
Skill 不是平铺的，有四个层级。先确定层级架构，再在每一层建立具体 Skill，最后定义层间协作规则。
3.2 四层 Skill 架构
Base Context · 始终注入 · Prompt Caching
不参与路由，每次对话开始时自动注入，是所有 Skill 的信息地基。
信号上下文 Skill order-context-and-signals
L0 · 路由层 · 每轮第一步
只做判断和路由，不解决业务问题，不写话术，不调用工具。
scope-guard（拒识 Skill） 服务覆盖范围表 4步路由决策树 拒识清单 边界陷阱
L1 · 场景层 · 最厚的 Skill
处理特定场景的完整问题，包含诊断-决策-执行三层，是 MECE 划分的覆盖主体。
履约场景（订单·履约中） 售后场景（订单·售后） 准时宝（产品专项） 商品券/神券（产品专项） 投诉抱怨 订单查询 其他功能
L2 · 执行层 · L1 的 function call
执行单一具体操作，不独立判断业务逻辑，由 L1 决定何时调用。
外呼 Skill（外呼商家/骑手） 转人工 Skill
横切层 · 叠加加载 · 由 L1 内部触发
正交于所有层级，在任何时刻提供沟通支持，不由路由层主动选择。
情绪管理 主动倾听 预期管理 降温技巧
3.3 建设优先级——分批次上线
第一批
上线的最小必要集合
原则：没有它上不了线
信号上下文
所有 Skill 的信息地基，缺失则信号集成全部失效
L0 边界 Skill
先定边界，模型才知道什么该处理、什么 reject
L1 履约场景
最高频场景约 40–60% 进线，缺失覆盖率严重不足
L1 售后场景
第二高频约 15–25% 进线，大量进线无处理
第二批
决定服务质量下限
原则：有了它质量更好，上线前完成
横切 情绪管理
遇 L3/L4 用户体验极差；框架可复用，调禁忌话术
L2 外呼操作
L1 Skill 中的外呼工具需要它作为执行规范
L2 转人工
必须在上线前决策：挽留型 vs 效率型
横切 主动倾听
防用户表达不完整时误判诉求
横切 预期管理
告知等待/无法解决等坏消息时需要
第三批
数据说有必要才建
原则：上线后根据指标决定
L1 产品专项
有特殊保险/权益产品且进线量可观时才建
L1 投诉抱怨
投诉场景升级率 > 20% 才值得单建
L1 订单查询
低频，可先在 L0 用自助入口处理
横切 降温技巧
有情绪管理后，看升级率决定是否需要
3.4 Skill 间冲突优先级
多个 Skill 同时加载时，必须有预定义的优先级规则，写在 System Prompt 层，不分散在各 Skill 内部。
优先级层级说明
最高System Prompt 硬约束输出格式、行为底线，任何情况不可违反
↓任何 Skill 中的 ⛔ 前置约束风控拦截、不可逆操作门控
↓L0 边界 Skill 的路由和拒识结论覆盖范围判定优先于业务处理
↓L1 场景 Skill 的业务判断主要的业务逻辑
↓L2 执行层 Skill 的操作规范单一操作的执行标准
最低横切层 Skill 的沟通建议话术和情绪应对，在业务逻辑确定后才生效
04
怎么写一个 Skill
每个 Skill 回答三个问题——「什么时候用我」「我知道什么」「我能做什么」——对应六个必备元素，按固定顺序组织。
4.2 六个必备元素
三个问题 → 六个元素：
什么时候用我 → ① 触发条件 + ② 反触发
我知道什么 → ③ 前置约束 + ④ 核心逻辑
我能做什么 → ⑤ 工具清单 + ⑥ 出口
①
触发条件
必须同时包含系统信号字段名和用户主诉特征，每条都能直接判断。
✅ 正确
当满足以下任一条件时激活：
· {订单是否超时} = 是
· {骑手轨迹异常} = 是
· 用户主诉：催单·问进度·未收到货
❌ 错误
当用户反馈配送问题时加载。
问题：「配送问题」无法直接判断
②
反触发（不覆盖什么）
明确说明不处理什么，并指向正确去处。只说覆盖什么、不说不覆盖什么，边界必然模糊。
✅ 正确
不覆盖：
· 餐品质量问题 → 售后 Skill
· 包装破损/撒漏 → 售后 Skill
· 准时宝规则咨询 → 产品专项 Skill
❌ 后果
模型遇到「外卖里有虫」时走履约路径，给出「我帮您催骑手」的错误回复。
③
前置约束 ⛔
所有硬性规则必须放在 Skill最开头，用 ⛔ 标注并编号。LLM 顺序读取，约束放结尾等于没有约束。
✅ 正确（前置，编号，说明原因）
⛔C1 诊断后才能决策，禁止跳过
⛔C2 促履约优先于退款
⛔C3 退款前必须得到用户明确正向回复
❌ 错误（约束后置）
（...2000字逻辑...

注意：以上退赔需先引导自助。
④
核心逻辑（三层分离）
诊断层（判责）、决策层（怎么做）、执行层（工具）必须分开写，不能混写。
✅ 正确（分层）
## 诊断层：谁出了问题？
义务清单·判责规则·信号读取

## 决策层：应该怎么做？
优先级总纲·失败模式应对

## 执行层：工具清单
❌ 错误（混写）
骑手超时→判为骑手责→退款→赔付
（判责和执行混在一段，模型逻辑混乱）
⑤
工具清单
所有工具必须用标准表格列出，四列缺一不可：工具名 / 触发条件 / 参数 / 关键约束。
✅ 正确（四列齐全）
| 工具名 | 触发条件 | 参数 | 约束 |
| 取消订单 | 非用户责+用户同意 | {责任方} | ⛔C3用户确认 |
| 操作赔付 | 订单已结束+同意 | {金额}{形式} | 进行中禁止 |
❌ 错误
工具表只有名称和触发条件，缺约束列→模型不知道取消订单需用户确认，直接执行。
⑥
出口
说明何时切换到其他 Skill，以及切换到哪里。没有出口，模型在诉求偏移时不知所措。
✅ 正确
| 用户提到少送/错送 | → 售后 Skill |
| 用户要转人工(第2次坚持) | → 转人工 Skill |
| 问准时宝赔付规则 | → 准时宝 Skill |
4.4 过度指定——不是越详细越好
过度指定：规定得太死，模型在边缘情况要么死板执行（错误结果），要么不知道该怎么做。区分三层：硬规则 / 软指引 / 留白。
层次定义退款告知示例
硬规则 ⛔ 写死，无例外，任何情况都必须执行 ⛔C3 执行取消订单前，必须得到用户明确正向回复（好/行/退吧），沉默不算同意
软指引 推荐做法，允许根据情境调整 退款告知推荐含：① 一句话说明原因 ② 一句话说明退款后不再配送
留白 明确不规定，交给模型根据上下文判断 用户犹豫时如何再次确认（每次情境不同，不宜固定话术）
自查问题：写完一条规则后问自己——「有没有一种明显应该例外的情况，但这条规则会强制执行到那种情况上？」有 → 过度指定了，考虑改为软指引或加例外条件。
05
各层级标准骨架
先读统一骨架（所有层级共用），再查差异表（各层级特有字段）。不需要在五份独立模板间来回对照。
5.0 统一骨架（所有层级通用）
统一骨架模板

5.1 各层级差异表

5.2 关键决策点——写作前必须拍板
L1
判责方法选哪种？
选项 A：义务清单遍历（外卖）
阶段 × 角色矩阵，逐项检查违约节点
优点：容错性高，信号缺失时也能推理
理论依据：清单决策模型（Gawande, 2009）
选项 B：优先级瀑布 P1–Pn（歪马）
信号命中第一条即停，快速判断
优点：速度快，逻辑清晰
理论依据：满意化决策（Simon, 1947）
判断依据：场景标准化程度 + 信号字段可靠性。不确定时优先选义务清单遍历——容错性更高。
L2
转人工策略选哪种？
选项 A：挽留型（外卖）
第一次要转 → 说明 AI 能力，提供替代方案
第二次坚持 → 才转接
适合：用户等待成本低，AI 覆盖率高
选项 B：效率型（歪马）
第一次要转 → 尝试一次并行解决
第二次坚持 → 无条件立即转接，不挽留
适合：用户时间敏感，AI 被用户不信任
判断方法：问一线客服——「挽留成功率多高？失败的挽留会不会让用户更烦？」失败率高且用户更烦 → 用效率型。

08
开放问题
以下问题没有标准答案，本章帮助理解为什么没有答案，以及实践中应该观察什么。
问题实践建议
什么指标说明 Skill 写得好？
🔲 开放问题		 先追踪「Skill 命中后转人工率」作为代理指标（容易追踪，升级率高=Skill 没解决问题），逐步建立更完整的评估体系。FCR 最能体现质量，但需要跨次进线追踪，复杂度高。
颗粒度怎么决定？
⚙️ 合理推断		 加载后超过 50% 内容与当前对话无关 → 这个 Skill 太粗了，考虑拆分。先粗后细，让数据告诉你哪个 Skill 升级率偏高，再针对性拆分。别在写作阶段反复纠结颗粒度。
Skill 边界重叠怎么处理？
🔲 开放问题		 建立「边界问题记录表」，把运行中发现的模糊案例记录下来，定期讨论形成约定。通常：更具体的优先于更宽泛的。
模型升级后 Skill 会失效吗？
⚙️ 合理推断		 可能失效：语言示例数量要求、分层严格程度。不太可能失效：触发条件明确性、约束前置原则、结构化组织。每次换模型时重跑所有测试用例。
