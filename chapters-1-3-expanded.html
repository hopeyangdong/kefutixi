Prompt 工程 2020—2023

💡 什么是 Prompt 工程？

Prompt 工程 = 通过精心设计输入文本（提示词），引导大模型生成高质量输出。

这是人类与大模型交互的第一种范式。你的全部输入就是 AI 唯一的信息来源——它不知道你有什么文档，不记得昨天聊了什么，也不能去查任何系统。你输入的那段话，就是它的整个世界。

📱
类比：给一个天才陌生人发短信。
这个人极其聪明，博览群书，但他只能看到你发的这一条短信。他不知道你是谁，不知道你在什么公司，不知道你的业务规则。回答质量 100% 取决于你这条短信写得有多清楚、多完整。

🚀 GPT-3 Few-Shot 的震撼：2020 年发生了什么？

要理解 Prompt 工程为什么是一场革命，我们得先看看2020 年之前，做一个客服 AI 有多痛苦。

假设你要做一个「退款原因分类器」——用户申请退款时，自动判断是「商品质量问题」「配送损坏」「口味不合」还是「其他」。在传统 NLP 时代，流程是这样的：

①
标注数据

找 20 个标注员，标注 10,000~50,000 条退款记录

②
训练模型

工程师用 BERT/TextCNN 训练一个分类模型，调参跑实验，周期 2-4 周

③
部署上线

模型封装成 API，部署到服务器，接入业务系统

④
持续维护

每季度补充新数据、重新训练、更新线上模型

整个流程下来，最少 2 个月、花费几十万，还需要一支包含标注员、算法工程师、后端工程师的技术团队。如果业务规则变了（比如增加了一个「配送超时」的退款类型），你得重新标注数据、重新训练——又是两周起步。

然后，2020 年 5 月，OpenAI 发布了 GPT-3。

📄
GPT-3 论文：Brown et al.《Language Models are Few-Shot Learners》
2020年5月 · OpenAI · 31位作者
arxiv.org/abs/2005.14165
首次系统展示 Few-Shot Prompting：在提示词中给几个示例，模型就能学会新任务，无需微调。

论文标题翻译过来就是：「语言模型是少样本学习者」。什么意思？你不用标注 1 万条数据了——在提示词里给 3 个例子，模型就能学会你的分类规则。

同样是退款原因分类，GPT-3 的做法：

## 退款原因分类 Prompt（Few-Shot） 你是一个客服退款原因分类助手。根据用户的退款描述，分类到以下类别之一： - 商品质量问题 - 配送损坏 - 口味不合 - 配送超时 - 其他 ## 示例： 用户描述：「汤全洒了，米饭也凉了」 分类：配送损坏 用户描述：「点的微辣结果巨辣，完全没法吃」 分类：口味不合 用户描述：「鸡腿只咬了一口发现是生的」 分类：商品质量问题 ## 请分类： 用户描述：「等了一个半小时才到，菜都凉透了」 分类：

就这么一段文字，没有训练、没有标注、没有部署——直接输入，GPT-3 就能输出「配送超时」。从 2 个月缩短到 2 分钟。

🎯 革命性在哪？不是说 GPT-3 比 BERT 准确率高多少（早期准确率其实不一定更高），而是门槛降到了零：不需要标注团队、不需要算法工程师、不需要 GPU 训练——一个运营人员，打几行字就能做到。这才是真正的范式转变。

🧠 Chain-of-Thought：让 AI 「展示推理过程」

📄
Chain-of-Thought 论文：Wei et al.《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》
2022年1月 · Google Research · NeurIPS 2022
arxiv.org/abs/2201.11903
发现在提示词中加入推理步骤，可以显著提升模型在数学、逻辑、常识推理任务上的表现。

Few-Shot 解决了「怎么让 AI 学会分类」，但有些任务不只是分类那么简单——它需要推理判断。比如客服退款审批：用户说「外卖少了一个菜」，该不该批准退款？退多少？

我们来看不加 CoT 和 加了 CoT 的巨大区别：

❌ 不加 Chain-of-Thought：直接猜答案

Prompt: 用户投诉「我点了3个菜，只收到2个，少了一份宫保鸡丁（28元）」，请判断是否应该退款。 AI 回复: 同意退款。

就一句话。你根本不知道它为什么同意——是看了订单吗？考虑了用户历史了吗？这个结论能信吗？万一是恶意退款呢？

✅ 加了 Chain-of-Thought：展示推理链

Prompt: 用户投诉「我点了3个菜，只收到2个，少了一份宫保鸡丁（28元）」。 请一步一步分析以下维度后给出结论： 1. 订单实际情况（点了几个菜、实收几个） 2. 商家/骑手责任判断 3. 用户历史退款记录是否异常 4. 推荐处理方案 AI 回复: 让我一步步分析： 1. 订单情况：用户下单 3 个菜品，实际收到 2 个，缺少「宫保鸡丁」（28元）。 属于典型的「少餐」问题。 2. 责任判断： - 少餐通常是商家漏装或骑手遗漏导致 - 需要看商家出餐记录。如果商家已出餐3份而骑手只送了2份 → 骑手责任 - 如果商家本身只出了2份 → 商家责任 - 无论哪种情况，用户都不应承担损失 3. 用户历史： - 如果用户近30天退款次数 ≤ 2 次 → 正常用户，直接退 - 如果用户近30天退款次数 ≥ 5 次 → 疑似异常，建议人工复审 4. 推荐方案： - 退还缺少菜品金额 28 元 - 如金额 ≤ 30 元且用户无异常记录 → 可自动审批 - 附带道歉话术 + 优惠券补偿

看到区别了吗？同样的问题，加了「一步一步分析」这几个字，AI 的回答从一句话变成了一份完整的分析报告。而且因为推理过程透明了，你可以检查它的逻辑是不是对的——如果第 3 步的判断标准不符合公司政策，你可以针对性地调整 prompt，而不是像之前那样面对一个黑盒。

🎯 CoT 的本质：不是让 AI 变聪明了，而是强迫 AI 把思考过程展开。就像让学生在数学考试时必须写解题步骤——一步一步写出来的过程中，自己会发现逻辑漏洞，最终答案的正确率自然更高。

🔑 5 种核心 Prompt 技巧详解

下面每种技巧都给出客服场景的完整、可直接使用的 Prompt 模板，不是一句话概括。

技巧 ① Zero-Shot（零样本）—— 不给任何示例，直接告诉 AI 干什么。

你是美团客服质检专员。请阅读以下客服对话记录，评估客服人员的服务质量。 评分标准（1-5分）： - 响应速度：是否在30秒内首次回复 - 问题解决：是否有效解决了用户问题 - 服务态度：用语是否礼貌、有同理心 - 流程规范：是否按标准流程操作 对话记录： [用户] 我的外卖都等了40分钟了还没到！ [客服] 稍等。 [客服] 帮你查了，骑手还在取餐。 [用户] 那要等到什么时候？？ [客服] 预计还要15分钟，你再等一下。 请输出 JSON 格式的评分结果，包含每项分数和总体评语。

适用场景：任务定义清晰、标准明确的场景。优点是简洁高效，缺点是如果 AI 对你的「标准」理解有偏差，输出就不稳定。

技巧 ② Few-Shot（少样本）—— 给 2-5 个示例，定义输入输出标准。

你是客服话术生成助手。根据用户的投诉内容，生成标准回复话术。 ## 示例： 投诉：外卖洒了 话术：非常抱歉给您带来不好的体验！您的餐品在配送过程中出现了洒漏情况，我已帮您申请该商品的全额退款，预计1-3个工作日到账。同时赠送您一张5元无门槛优惠券，期待下次为您提供更好的服务～ 投诉：送错地址了 话术：非常抱歉，核实到您的订单确实配送到了错误地址。我已联系骑手为您重新配送/为您安排全额退款（预计1-3个工作日到账）。对此次失误深感抱歉，已赠送您一张8元优惠券作为补偿！ 投诉：商品和图片不一致 话术：很抱歉给您造成了困扰！已核实您反馈的情况，商家出餐与菜单展示确实存在差异。我已帮您申请差价退款XX元，同时会将情况反馈给商家进行整改。感谢您的反馈，祝您用餐愉快～ ## 请生成： 投诉：骑手态度恶劣，打电话催我下楼还骂人 话术：

适用场景：需要统一风格和格式的场景。示例就是最好的说明书——与其用文字描述「回复要有同理心、要给解决方案、要给补偿」，不如直接给 3 个标杆回复。

技巧 ③ Chain-of-Thought（思维链）—— 要求 AI 先推理再结论。

你是客服退款审批助手。请按以下步骤分析退款申请： Step 1: 提取关键信息（订单号、商品、金额、投诉内容） Step 2: 判断责任方（商家/骑手/用户/平台） Step 3: 检查退款政策（该情况是否符合自动退款条件） Step 4: 评估风险（用户是否有异常退款行为） Step 5: 给出结论（退款金额 + 补偿方案 + 置信度） 退款申请：用户称「收到的奶茶全是冰的，我备注了去冰，喝了一口就肚子疼」

适用场景：需要多因素综合判断的决策场景。强制分步能避免 AI 跳跃式得出结论。

技巧 ④ 角色设定（Persona）—— 给 AI 一个专业身份，激活特定知识。

你是一个拥有10年美团客服管理经验的培训师。你现在要培训一批新入职的客服人员。 你的特点： - 说话接地气，用真实场景举例 - 特别擅长处理「用户情绪激动」的场景 - 总是先肯定新人做对的地方，再指出改进空间 请对以下新客服的回复做点评和改进建议： 用户消息：「你们是不是骗子公司？！我退款三天了还没到账！再不退我就投诉到12315！」 新客服回复：「您好，退款需要1-3个工作日处理，请您耐心等待。」

适用场景：需要特定视角或专业深度的场景。角色设定不是「玩角色扮演」，而是通过设定身份来约束 AI 的输出风格、知识范围和回答深度。

技巧 ⑤ 格式约束（Output Formatting）—— 规定输出结构，方便下游系统对接。

分析以下客服对话，输出严格的 JSON 格式结果。 要求的 JSON 结构： { "issue_type": "退款|配送|投诉|咨询", "urgency": "高|中|低", "sentiment": "愤怒|不满|中性|满意", "requires_human": true/false, "suggested_action": "具体建议操作", "confidence": 0.0-1.0 } 对话内容： [用户] 骑手把餐放在门口就走了，也不敲门也不打电话，我两小时后才发现，饭菜全凉了 [用户] 而且我备注了「请打电话」，骑手根本没看备注 请直接输出 JSON，不要其他解释。

适用场景：AI 输出需要被程序读取的场景。你的客服工单系统要根据 AI 的分析结果自动分配工单、设置优先级——那 AI 输出必须是结构化的，不能是一段自然语言。

⚠️ Prompt 的三大天花板：为什么光靠写提示词不够？

Prompt 工程很强大，但用着用着你会撞上三面墙，怎么写都绕不过去。

天花板 ①：知识塞不进去

你的客服知识库有 500 页，涵盖退款政策、配送规则、商家管理、活动规则、特殊品类处理（药品、鲜花、生鲜各一套规则）……GPT-3.5 的上下文窗口只有 4K token ≈ 3000 字，也就是大约 3 页 A4 纸。就算用 GPT-4 的 128K 窗口，也只能塞进约 100 页——而且全塞进去，AI 在「大海捞针」式的长文本中准确率会大幅下降。

具体场景：用户问「我买的药品拆封了还能退吗？」这个答案在知识库第 347 页。你怎么办？把 500 页全粘进去？token 不够。只粘第一章？答案不在那里。你需要一种方式自动找到第 347 页那段话——这就是第二章要讲的 RAG。

天花板 ②：每次对话都要重新教

昨天你花了 20 分钟，通过反复调试，教会了 AI 你们公司的退款审批流程——什么情况自动退、什么情况要人审。今天打开一个新对话，AI 全忘了。你得重新教一遍。

更要命的是，如果你的客服团队有 50 个人在用 AI，每个人都要各自教一遍。有人教得好，有人教得差，AI 的表现参差不齐——没有统一的知识基座。

天花板 ③：AI 只能说，不能做

AI 能告诉你「这个订单应该退款 28 元」，但它不能真的去操作退款。它不能查询你的订单数据库，不能调用退款接口，不能更新工单状态。它只是一个「说话的人」，不是一个「干活的人」。

客服场景里，用户要的不是一个分析报告，而是问题被解决。「帮我退款」不是要你说「我建议退款」，而是要你真的点那个按钮把钱退了。

天花板客服场景后续解决方案
知识塞不进500页知识库放不进 prompt→ 第2章 RAG 检索增强
没有记忆每次对话重新教规则→ 第2章 Memory 记忆机制
不能操作只能说建议，不能执行退款→ 第3章 Workflow / 第4章 Agent

📊 Andrew Ng 的实测数据：迭代 > 换模型

📄
Andrew Ng《Agentic Design Patterns Part 1》· 2024年3月20日
deeplearning.ai
斯坦福教授、DeepLearning.AI 创始人，用 HumanEval 编码基准做了对比实验。

Andrew Ng 做了一个特别有说服力的实验——用编程任务（HumanEval）对比三种方式的表现：

方式准确率背后的逻辑
GPT-3.5 Zero-Shot（纯 Prompt）48.1%写一次 prompt，AI 输出一次答案，不迭代
GPT-4 Zero-Shot（纯 Prompt）67.0%换了更强的模型，但仍然是一次性输出
GPT-3.5 + Agent 迭代循环95.1%弱模型，但加了「写代码→跑测试→看报错→改代码」的迭代循环

看到那个关键数字了吗？弱模型 + 迭代流程（95.1%）碾压了强模型的一次性输出（67.0%）。差距不是一点半点，是 28 个百分点。

这告诉我们什么？

🎯 Andrew Ng 的结论：与其花大钱把 GPT-3.5 升级到 GPT-4，不如在流程上做文章——让 AI 能自我检查、自我纠错、反复迭代。这就是为什么行业从 Prompt 工程出发，一路往 Workflow → Agent → Harness → Loop 演进。

换个客服场景来理解：一个普通客服 + 完善的质检流程（写回复→质检打分→不合格重写→再质检）的效果，很可能比一个资深客服不做质检直接发出去要好。流程制度的力量，大于个人能力的差距。

🧪 小测验 1-A

你想让 AI 分析一份 100 页的客服手册，用 Prompt 工程能搞定吗？

A. 能，把 100 页全粘贴进去就行
B. 能，分 10 次粘贴，每次 10 页
C. 不太行，需要上下文工程（RAG）来解决
✅ 正确！100 页远超早期模型的 token 限制；即便用大窗口模型全塞进去，准确率也会下降。分批粘贴则丢失跨批上下文。这正是上下文工程（RAG）要解决的问题——让 AI 动态检索所需段落，而不是全塞进去。

🧪 小测验 1-B

Andrew Ng 的实验显示，GPT-3.5 + Agent 迭代循环的准确率（95.1%）超过了 GPT-4 Zero-Shot（67.0%）。这说明什么？

A. GPT-3.5 其实比 GPT-4 更强
B. 好的流程/迭代机制比单纯换更强的模型更有效
C. Prompt 工程已经没用了
✅ 正确！这不是说 GPT-3.5 比 GPT-4 强——而是说「弱模型 + 好流程」优于「强模型 + 无流程」。Prompt 工程仍然是基础技能，但要做出优秀的 AI 应用，还需要在迭代、检验、纠错机制上下功夫。这就是后面几章要讲的内容。

上下文工程 2023—2025

💡 从 Prompt 到 Context：一次认知升级

上下文工程 = 在 AI 回答之前，系统地管理它能「看到」的全部信息。

第 1 章我们讲了 Prompt 工程——怎么写提示词。但到了实际业务中你会发现，问题往往不在于「你问得对不对」，而在于「AI 手上有没有足够的信息来回答」。

📝
类比：考试。
Prompt 工程 = 考试时怎么审题、怎么答题（技巧层面）。
上下文工程 = 考试前准备什么复习资料（信息层面）。

一个学生审题技巧再好，如果考试范围的内容他根本没复习到，也答不出来。同理，prompt 写得再精巧，如果 AI「看不到」你的客服知识库、看不到订单数据、不知道历史对话——它就只能靠通用知识瞎猜。

所以，上下文工程的核心问题不是「怎么问」，而是——

🎯 核心问题：在 AI 处理每个请求时，它「看到」的信息是不是对的、够不够、有没有噪声？

📎 这个词是怎么来的？

🐦
「Context Engineering」术语来源：Tobi Lütke（Shopify CEO）
2025年6月19日 · X/Twitter
原文：「I really like the term 'context engineering' over prompt engineering. It describes the core skill better.」
翻译：「比起 prompt engineering，我更喜欢 context engineering 这个说法。它更准确地描述了核心技能。」

Andrej Karpathy（OpenAI 联合创始人、前 Tesla AI 总监）于 6月23日回复「+1 for context engineering」表示认可。
📄
Anthropic 权威定义：《Effective Context Engineering for AI Agents》
2025年9月29日
anthropic.com
原文：「Context engineering is about considering the holistic state available to the LLM at any given time and what potential behaviors that state might yield.」

我们来逐句拆解 Anthropic 的这段定义：

  • 「holistic state」（全局状态）—— 不只是 prompt 本身，而是 AI 能看到的所有信息：系统指令 + 用户消息 + 检索到的文档 + 工具描述 + 历史对话 + 记忆摘要……整个信息包。
  • 「available to the LLM at any given time」（在任何给定时刻）—— 强调动态性。不同的用户提问，检索出的文档不同；不同的对话阶段，加载的记忆不同。上下文是动态组装的，不是写死的。
  • 「what potential behaviors that state might yield」（这个状态可能产生什么行为）—— 你给 AI 什么信息，它就做什么事。如果上下文里有退款按钮的 tool 描述，它就可能去退款；如果没有，它就只能告诉用户「请联系人工客服」。上下文决定行为。

维度 ① CoT 思维链：教 AI 怎么想

第 1 章已经讲过 CoT，但在上下文工程视角下，CoT 不只是一个 prompt 技巧——它是往上下文中注入推理模板。比如你在系统指令中内置一套退款判断的思维链模板，所有用户提问都会自动触发这套分析流程，而不需要每个用户每次都手动加「请一步一步分析」。

维度 ② RAG 检索增强：给 AI 看什么

这是最重要的维度，下面会详细展开。

维度 ③ Memory 记忆：让 AI 记住什么

Prompt 工程的天花板之一就是「每次对话都要重新教」。Memory 解决这个问题——把重要信息持久化存储，新对话自动加载。

## Memory 在客服场景的例子 短期记忆（同一次对话内）： 用户先说「我想退款」，又说「订单号是 MT20241201001」 → AI 需要记住前面说的「退款」意图，结合订单号一起处理 长期记忆（跨对话持久化）： 这个用户上次投诉过骑手态度问题，我们赠送了优惠券 → 这次用户又来投诉，AI 能看到历史记录，知道这是「二次投诉」 → 自动升级到 VIP 处理通道，而不是再发一张优惠券敷衍

维度 ④ Tool 描述：告诉 AI 能用什么

这个最容易被忽视。当你给 AI 配备了「查询订单」「发起退款」「发优惠券」等工具时，你需要在上下文中准确描述每个工具的用途和使用条件。描述写得好，AI 才知道什么时候该调用哪个工具。

## Tool 描述示例 工具名称：query_order 功能：根据订单号查询订单详情（包括商品、金额、状态、配送信息） 参数：order_id (string, 必填) 什么时候用：当用户提到具体订单号，或需要核实订单信息时调用 什么时候不用：用户只是一般性咨询（如「你们的退款政策是什么」），不涉及具体订单时不要调用 工具名称：process_refund 功能：对指定订单发起退款 参数：order_id (string, 必填), amount (number, 必填), reason (string, 必填) 什么时候用：已确认用户投诉属实，且符合退款条件时调用 什么时候不用：用户还未提供订单号，或投诉尚未核实时绝对不能调用

🔍 RAG 完整工作流详解

📄
RAG 原始论文：Lewis et al.《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》
2020年5月 · Facebook AI / UCL / NYU · NeurIPS 2020
arxiv.org/abs/2005.11401
首次提出 RAG 架构：检索外部知识 + 生成回答，解决模型知识过时和幻觉问题。

RAG 不是一个简单的「搜索+回答」，它有 5 个关键步骤。我们用一个完整的客服场景走一遍：

假设你有一份 500 页的客服知识手册，涵盖退款政策、配送规则、商家管理、活动规则等等。

Step 1：文档切片（Chunking）

把 500 页手册切成小段落。为什么要切？因为用户问「骑手超时怎么赔」，你不需要给 AI 看全部 500 页，只需要「配送超时赔偿政策」那一段。

## 切片示例 原始文档（第 127 页）： 「5.3 配送超时赔偿规则 5.3.1 超时定义：实际送达时间超过预估送达时间15分钟以上视为超时。 5.3.2 赔偿标准： - 超时15-30分钟：赔偿订单金额的10%，最低3元 - 超时30-60分钟：赔偿订单金额的20%，最低5元 - 超时60分钟以上：可申请全额退款 5.3.3 特殊说明：恶劣天气（暴雨/暴雪/台风）期间，超时阈值调整为30分钟。」 → 切成一个 chunk（片段），保留章节标题作为元数据

500 页手册大约切成 2000 个 chunk，每个 chunk 200-500 字。

Step 2：向量化（Embedding）

每个 chunk 通过 Embedding 模型转换成一个向量（一串数字）。这个向量代表了这段文字的「语义含义」。语义相近的文字，向量在空间中的距离就近。

🧭
类比：给每段文字标注 GPS 坐标。
「配送超时赔偿」和「骑手迟到补偿」虽然用词不同，但语义相近，所以它们的「GPS 坐标」很近。搜索时，只要用户问题的坐标在附近，就能找到它们。这就是为什么向量检索比关键词搜索强——它理解「意思」，而不只是匹配「字面」。

Step 3：存入向量数据库

把 2000 个 chunk 的向量存入向量数据库（如 Pinecone、Milvus、Weaviate）。这就像建了一个「语义搜索引擎」。

Step 4：用户提问时检索 Top-K

用户问「骑手超时了怎么赔付？」→ 系统把这句话也转成向量 → 在向量数据库中找到最近的 3-5 个 chunk。

## 检索结果示例 用户提问：「骑手超时了怎么赔付？」 检索到的 Top-3 chunk： Chunk #1（相似度 0.94）： 「5.3 配送超时赔偿规则 ... 超时15-30分钟：赔偿订单金额的10%...」 Chunk #2（相似度 0.87）： 「5.3.3 特殊说明：恶劣天气期间，超时阈值调整为30分钟...」 Chunk #3（相似度 0.82）： 「5.4 配送异常处理流程 ... 用户可通过APP内「申请赔付」入口...」

Step 5：拼装 Prompt 送给 LLM

把检索到的 chunk + 用户问题 + 系统指令拼在一起，组成一个完整的 prompt：

System: 你是美团客服助手。请仅根据以下参考资料回答用户问题。 如果参考资料中找不到答案，请说「抱歉，我需要转接人工客服为您处理」。 回答后请标注信息来源的章节号。 参考资料： --- [来源: 第5.3节] 配送超时赔偿规则...超时15-30分钟：赔偿订单金额的10%... [来源: 第5.3.3节] 特殊说明：恶劣天气期间，超时阈值调整为30分钟... [来源: 第5.4节] 配送异常处理流程...用户可通过APP内「申请赔付」入口... --- 用户问题：骑手超时了怎么赔付？
一句话总结 RAG：不是把 500 页全塞给 AI，而是只给它最相关的几段。就像你不会把整个图书馆搬给朋友，而是帮他找好那几页，翻到那一段，指着说「看这里」。

💣 RAG 的三个常见坑

RAG 不是银弹。做得不好，效果可能比纯 Prompt 还差。以下是三个最常见的坑：

坑 ①：切片太大 → 噪声太多

如果你把整个「第五章 配送管理」（20 页）作为一个 chunk，用户问「超时怎么赔」时，AI 要在 20 页里自己找答案——大量无关信息（正常配送流程、骑手调度规则）会干扰它的判断。

客服场景：用户问「少餐怎么退款」，你给 AI 的 chunk 包含了整个「退款章节」——里面有正常退款、商品质量退款、配送问题退款、恶意退款处理等十几种场景。AI 可能把「恶意退款」的处理方式混进回答里，告诉用户「您的退款需要提供视频举证」——其实少餐根本不需要举证，直接退就行。

坑 ②：切片太小 → 丢失上下文

如果每个 chunk 只有一两句话，比如只切到「超时30-60分钟：赔偿订单金额的20%」，丢掉了前面的「超时定义」和后面的「恶劣天气特殊规则」——AI 不知道什么叫超时，也不知道暴雨天标准不一样。

客服场景：用户在暴雨天投诉超时 20 分钟。AI 检索到了「超时15分钟以上赔偿」这个 chunk，但没检索到「恶劣天气阈值调整为30分钟」那个 chunk。结果 AI 告诉用户可以赔付——但按照公司政策，暴雨天 20 分钟根本不算超时。这就是切太碎导致上下文缺失。

坑 ③：检索不准 → 答非所问

向量检索是基于语义相似度的，有时候语义相似的内容并不是用户真正要问的。

客服场景：用户问「怎么取消退款申请？」（用户不想退了，想把退款撤回来）。向量检索可能返回的是「怎么申请退款」的章节——因为「取消退款」和「申请退款」在语义上很接近（都和退款相关），但用户要的是完全相反的操作。AI 如果基于错误的 chunk 回答，就会教用户「第一步打开退款页面，第二步选择退款原因…」——完全牛头不对马嘴。
常见坑原因解法方向
切片太大噪声信息干扰 AI 判断按语义段落切片，保持 200-500 字
切片太小丢失必要上下文重叠切片（overlap）+ 保留章节元数据
检索不准语义相似 ≠ 意图匹配混合检索（向量+关键词）+ 重排序（Reranker）

🆚 同一个问题，两种做法的对比

我们用一个具体场景来看 Prompt 工程和上下文工程的做法差异：

场景：用户问「我买的药品已经拆封了，能退款吗？」

Prompt 工程的做法：

你是美团客服。请回答：用户买的药品已经拆封了，能退款吗？ AI 回复：一般来说，药品属于特殊商品，拆封后通常不支持退款， 因为涉及到药品安全和卫生问题。建议您联系商家协商处理。 → 这是 AI 基于通用知识猜的，不一定对。你们公司可能有特殊政策。

上下文工程的做法：

System: 你是美团客服助手。仅根据以下参考资料回答。 参考资料（RAG 检索结果）： [来源: 第8.2.1节 药品退款政策] 「药品类商品退款规则： 1. 未拆封：支持7天无理由退款 2. 已拆封但存在质量问题（如过期、变质、错发）：支持全额退款 3. 已拆封且无质量问题：不支持退款 4. 特殊情况：如用户能提供医生证明该药品不适合使用，可申请退款审核」 用户问题：我买的药品已经拆封了，能退款吗？ AI 回复：根据我们的药品退款政策（第8.2.1节），已拆封的药品分两种情况： 1. 如果药品存在质量问题（过期、变质、或发错药了），可以全额退款 2. 如果药品没有质量问题，不支持退款 请问您的药品是出现了什么问题呢？如果存在质量问题，我可以马上帮您处理退款。 → 回答准确、有依据、可溯源到具体章节号，还主动追问了一步。
对比维度Prompt 工程上下文工程
关注点怎么写提示词AI 能看到什么信息
知识来源模型训练数据（可能过时或不准）你的知识库（实时、准确）
能处理的信息量受限于上下文窗口理论上无上限（检索后只取片段）
是否可溯源不能，可能是 AI 编的可以，标注出处章节号
实时性截止到模型训练日期知识库随时更新即可
一致性每个人写的 prompt 不同，答案不同统一知识库，回答一致
维护成本每次对话手动粘贴规则维护知识库即可，自动检索

📖 Anthropic 的完整定义解读

📄
Anthropic《Effective Context Engineering for AI Agents》· 2025年9月29日
anthropic.com

Anthropic 在这篇文章中给出了一段精炼的定义，我们来逐句翻译和解读：

原文：「Context engineering is about considering the holistic state available to the LLM at any given time and what potential behaviors that state might yield. It includes things like what it knows about the world (from retrieval / RAG), how it should approach problems (from system prompts / instructions), what tools it can use (from tool definitions), and what it remembers about the current interaction (from conversation history and memory).」

翻译拆解：

  • 「holistic state（全局状态）」—— 上下文工程看的不是某一个维度，而是 AI 能接触到的全部信息的总和
  • 「what it knows about the world（它对世界的认知）」—— 对应 RAG，从你的知识库检索来的信息
  • 「how it should approach problems（它应该怎么处理问题）」—— 对应 系统指令和 CoT，定义推理方法
  • 「what tools it can use（它能用什么工具）」—— 对应 Tool 描述
  • 「what it remembers（它记住了什么）」—— 对应 Memory，对话历史和长期记忆

注意到了吗？这正好对应我们前面讲的四个维度。Anthropic 的定义就是把这四个维度统一成了一个框架。

🎯 一句话总结：上下文工程 = 动态管理 AI 在每一刻能看到的全部信息。不是写一个好的 prompt 就完事了，而是搭建一个完整的信息供给系统——检索该看的文档、加载该用的记忆、描述该调的工具、注入该走的思路——然后把这些动态组装成 AI 的「输入」。

A. 切片太大，噪声太多
B. 切片太小，丢失了上下文

✅ 正确！「恶劣天气特殊规则」和「超时赔偿标准」在原文中可能是紧挨着的两段。如果切得太碎，它们被分到不同的 chunk，检索时只命中了一个，就丢失了关键上下文。解法是用重叠切片（overlap）或者把逻辑上相关的段落保持在同一个 chunk 里。

Workflow 工作流 2024

💡 为什么要从「一次调用」走向「多步编排」？

Workflow = 通过预定义的代码路径，编排多个 LLM 调用和工具操作。

关键词是「预定义」——执行路径是人类设计好的。AI 只在每个节点上做具体决策，但不能自主改变流程走向。

先说为什么需要它。我们来看一个真实的客服场景：

用户投诉：「我的外卖少了一个菜，而且汤还洒了一半，骑手态度还特别差，让我自己看着办。我要退款！还要投诉这个骑手！」

这一条消息里，包含了至少 3 个问题：① 少餐 ② 汤洒了 ③ 骑手态度差。处理这个投诉，一次 LLM 调用搞不定——你需要：

①
问题拆解

识别出3个问题：少餐+洒漏+骑手态度

②
查询订单

调接口看订单详情、配送记录

③
分别判责

少餐→商家责任
洒漏→骑手/商家
态度差→骑手责任

④
计算赔偿

少餐退菜品金额 + 洒漏部分退款

⑤
处理投诉

骑手态度投诉走单独流程

⑥
生成话术

组合所有处理结果，生成一段完整的回复

而且每一步都可能走不同的分支——比如第③步，如果判定洒漏是包装问题，那是商家责任；如果是骑手颠簸导致的，那是骑手责任。不同的责任方，赔偿从不同的渠道出。

🏭
类比：流水线工厂。
每个工位上有一个工人（AI），工人很能干，但他只负责自己工位上的事。工位的顺序、转运规则（这个零件接下来去哪个工位）都是工厂设计好的。工人不能自己决定「我觉得这个零件应该跳过下一个工位直接去组装」——那是工厂的事。

这就是 Workflow 的价值：把一个复杂任务拆成多个步骤，每步由 AI 处理，但步骤之间的流转逻辑由人设计好。

📎 权威出处

📄
Anthropic《Building Effective Agents》· 2024年12月19日
anthropic.com
原文：「Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents, on the other hand, are systems where LLMs dynamically direct their own processes and tool usage.」

翻译：Workflow 是通过预定义代码路径编排 LLM 和工具的系统。Agent 则是 LLM 自主动态控制自身流程和工具使用的系统。

这是行业首次清晰区分 Workflow ≠ Agent。这篇文章发布后被整个 AI 行业广泛引用。

🧩 Anthropic 定义的五种 Workflow 模式（详解版）

Anthropic 在文中定义了五种 Workflow 设计模式。下面每种模式都用客服场景的完整例子来说明。

模式 ① Prompt Chaining（提示链）—— 串行流水线

A 的输出是 B 的输入，B 的输出是 C 的输入。就像流水线，一步接一步。

## 客服场景：处理外语用户的投诉 Step 1 - 语言检测 + 翻译 Input: 用户消息（可能是英文、日文、韩文） LLM 任务: 检测语言 → 翻译成中文 Output: 中文版投诉内容 ↓ 传递给下一步 Step 2 - 情绪分析 Input: 中文版投诉内容 LLM 任务: 分析用户情绪（愤怒/不满/中性）+ 紧急程度 Output: { emotion: "愤怒", urgency: "高" } ↓ 传递给下一步 Step 3 - 问题分类 Input: 中文投诉 + 情绪/紧急度 LLM 任务: 分类（退款/配送/商品质量/骑手投诉/其他） Output: { category: "配送问题-超时" } ↓ 传递给下一步 Step 4 - 生成回复 Input: 分类结果 + 情绪 + 原始投诉 LLM 任务: 按模板生成对应回复（注意匹配情绪等级调整语气） Output: 一段专业的客服回复（用用户的原始语言输出）

适用场景：每个步骤都是独立的、有明确输入输出的任务。好处是每步都可以单独验证和优化——翻译不准就调翻译那步的 prompt，不影响其他步骤。

模式 ② Routing（路由分发）—— 一个入口，多个出口

先识别输入的类型，再分发到不同的处理流程。就像医院的分诊台。

## 客服场景：统一入口，智能分流 用户消息进入 → LLM 分类器判断类型 ├── 退款问题 → 退款处理流程（查订单→判责→计算金额→执行退款） ├── 配送问题 → 物流查询流程（查配送状态→联系骑手→给用户反馈） ├── 商品质量 → 质量投诉流程（记录问题→联系商家→赔偿方案） ├── 骑手投诉 → 骑手管理流程（记录投诉→升级主管→反馈处理结果） └── 一般咨询 → 知识库问答（RAG 检索→回答）

适用场景：输入类型多样，每种类型有完全不同的处理逻辑。路由分发是客服场景中最常用的模式——因为客服本来就是一个「入口统一，问题多样」的场景。

模式 ③ Parallelization（并行处理）—— 同时做多件事

同一个输入，同时送给多个 LLM 处理不同维度，最后汇总。

## 客服场景：投诉责任判定（三方并行） 用户投诉：「外卖到了但是菜都洒了」 同时启动 3 个 LLM 分析： LLM-A 分析商家责任： → 检查商家打包是否规范（有没有用防洒包装） → 结论：打包规范，非商家责任 LLM-B 分析骑手责任： → 检查配送轨迹（有无异常绕路、骑行速度是否过快） → 结论：骑手骑行速度超标，存在急刹车记录，骑手责任 LLM-C 分析平台责任： → 检查是否存在系统派单不合理（距离过远、接单量过多） → 结论：派单正常，非平台责任 汇总：骑手责任 → 从骑手账户扣款赔偿用户

适用场景：需要多角度、独立评估的决策。并行的好处不只是快——更重要的是每个分析独立进行，避免了一个维度的结论影响另一个维度的判断。

模式 ④ Orchestrator-Workers（编排-执行）—— 一个老板带多个工人

一个「主管 LLM」理解总任务后，拆成子任务分配给多个「工人 LLM」。

## 客服场景：分析一条差评并制定改进方案 差评内容：「份量太少了，这个价格就给这么点东西？而且米饭还是 冷的。送餐速度倒是挺快的，但是包装太简陋，汤差点就洒了。」 Orchestrator（主管LLM）分析后拆成4个子任务： → Worker-1：情感分析 「整体负面（3/10），但有一个正面点（配送速度）」 → Worker-2：问题归类 「① 份量少（商品问题）② 米饭冷（出餐问题）③ 包装简陋（包装问题）」 → Worker-3：竞品对比 「同品类同价位商家平均份量是...，该商家确实偏少」 → Worker-4：改进建议 「建议商家：① 增加份量或降价 ② 出餐后保温 ③ 升级包装」 Orchestrator 汇总所有 Worker 的结果 → 输出完整的差评分析报告

适用场景：任务太大、太复杂，单次 LLM 调用处理不好；需要拆解后各个击破。注意与 Parallelization 的区别——并行是「同一个问题多角度看」，编排-执行是「大问题拆成小问题」。

模式 ⑤ Evaluator-Optimizer（评估-优化）—— 写完改、改完查、查完再改

一个 LLM 生成内容，另一个 LLM 评估质量，不合格就打回重做。

## 客服场景：高质量话术生成 Round 1: Generator（生成器LLM）： 用户投诉骑手态度差 → 生成回复： 「您好，很抱歉给您带来不好的体验。我们会对骑手进行处理。」 Evaluator（评估器LLM）打分： - 同理心表达：3/5（「很抱歉」太敷衍，没有具体回应用户的感受） - 解决方案：2/5（「进行处理」太模糊，用户不知道会怎么处理） - 补偿方案：0/5（完全没有提到任何补偿） - 总分：5/15 → 不合格，打回重写 - 改进建议：① 具体描述理解用户的感受 ② 说明具体处理措施 ③ 给出补偿方案 Round 2: Generator 根据反馈重写： 「非常理解您的感受，骑手的态度确实让人不舒服，这完全不应该发生。 我已经将此骑手的行为记录在案，会由配送主管在24小时内进行约谈处理， 并在3个工作日内将处理结果反馈给您。为了弥补这次不好的体验， 我为您发放了一张10元无门槛优惠券，已自动放入您的卡券包中。 再次向您道歉，感谢您的反馈帮助我们改进服务！」 Evaluator 再次打分： - 同理心表达：5/5 ✅ - 解决方案：5/5 ✅ - 补偿方案：4/5 ✅ - 总分：14/15 → 合格，输出

适用场景：对输出质量要求高的场景。这个模式本质上就是一个自动化的质检流程——跟你们人工客服的「写回复→质检打分→不合格重写」是一样的逻辑，只不过质检员也是 AI。

💻 代码实现：Routing 模式的伪代码

我们用 Python 伪代码展示一个 Routing（路由分发）的实现，让你感受一下 Workflow 在代码层面长什么样：

import llm # 假设的 LLM 调用库 def handle_customer_message(user_message): # Step 1: 用 LLM 做分类（路由节点） category = llm.classify( prompt=f"""将用户消息分类到以下类别之一，只输出类别名： - 退款问题 - 配送问题 - 商品质量 - 骑手投诉 - 一般咨询 用户消息：{user_message}""" ) # Step 2: 根据分类结果，走不同的处理流程（预定义路径） if category == "退款问题": return refund_workflow(user_message) elif category == "配送问题": return delivery_workflow(user_message) elif category == "商品质量": return quality_workflow(user_message) elif category == "骑手投诉": return rider_complaint_workflow(user_message) else: return faq_workflow(user_message) # RAG 问答 def refund_workflow(user_message): # 退款子流程（也是预定义的多步骤） order_id = llm.extract(user_message, field="order_id") order_info = database.query_order(order_id) # 查数据库 analysis = llm.analyze( prompt=f"""根据订单信息判断是否应该退款： 订单信息：{order_info} 用户投诉：{user_message} 请分析责任方和建议退款金额。""" ) if analysis.should_refund and analysis.amount <= 30: refund_api.process(order_id, analysis.amount) # 自动退款 return llm.generate_reply(analysis, tone="empathetic") else: return escalate_to_human(order_id, analysis) # 转人工

注意看这段代码的结构：每个 if-elif-else 分支都是人类预先定义好的。LLM 只在两个地方做决策：① 分类用户消息 ② 分析退款是否合理。但「退款问题走退款流程、配送问题走配送流程」这个路由规则，是写死在代码里的。

这就是 Workflow 和 Agent 的本质区别。

⚠️ 本章最重要的点：Workflow ≠ Agent

这是整个课程中最容易搞混、也最重要的一个区分。很多人在讨论「AI Agent」时，实际上说的是 Workflow。让我们把区别讲清楚。

📄
Anthropic 原文（2024年12月）：
「When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed. This might mean not building agentic systems at all.」

翻译：构建 LLM 应用时，我们建议找到最简单的解决方案，只在必要时才增加复杂度。这可能意味着根本不需要构建 Agent 系统。
维度Workflow（工作流）Agent（智能体）
谁决定下一步做什么？人类预定义的代码逻辑AI 自己决定
路径是否可预测？✅ 是，流程图是固定的❌ 否，每次运行可能不同
能处理意外情况吗？❌ 只能走预设分支✅ 可以即兴应对
成本可控吗？✅ 可预估 token 消耗⚠️ 可能反复试错，消耗不可控
出错时好排查吗？✅ 哪一步出错一目了然⚠️ 中间过程是黑盒
客服场景例子固定流程：分类→查单→判责→退款AI 自己决定先查单还是先问用户

打个比方：

🏨
Workflow 就像酒店前台。有一本《服务手册》——客人要订房，翻到第 3 章；客人要退房，翻到第 7 章；客人投诉，翻到第 12 章。手册上怎么写，就怎么做。

Agent 就像私人管家。你说「帮我安排明天的行程」，管家自己去查天气、查景点、订餐厅、叫车——他自己决定做什么、以什么顺序做、做到什么程度。
Anthropic 的核心建议（也是本章的核心主旨）：

「Workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed at scale.」

翻译：对于规则明确的任务，Workflow 提供更好的可预测性和一致性；只有在需要灵活性和模型驱动决策的场景，才应该考虑 Agent。

一句话：能用 Workflow 解决的，别上 Agent。

🔮 什么时候该用 Workflow？什么时候该上 Agent？

这里给你一个实用的判断框架：

判断条件适合 Workflow需要 Agent
处理流程流程固定，可以画出流程图流程不确定，需要根据情况随机应变
输入类型可以穷举（退款/配送/投诉/咨询…）无法穷举，千奇百怪的请求
质量要求每次结果要高度一致允许一定的创造性和变化
风险等级出错代价大（涉及退款/财务）出错可接受（如生成初稿）
执行步骤通常 3-8 步，步骤明确步骤数不确定，可能需要 20+ 步

在客服场景中，绝大多数任务适合 Workflow：

  • ✅ Workflow：退款审批流程、工单分类分发、常规问题自动回复、质检打分
  • ✅ Workflow：配送超时赔偿计算、新用户引导流程
  • ⚠️ Agent：处理从未见过的新型投诉（比如用户投诉「骑手用我的电话号码注册了借贷平台」）
  • ⚠️ Agent：需要跨系统、多轮交互、动态调整策略的复杂问题
🎯 实操建议：先把 80% 的常见问题用 Workflow 自动化（投入产出比最高），剩下 20% 的复杂/边缘 case 再考虑是用 Agent 还是转人工。Anthropic 的建议非常实在——不要为了「用上 Agent」而用 Agent，简单方案能解决的问题，简单方案永远是最好的方案。

🧪 小测验 3-A

用户发消息给客服 AI，系统先分类问题类型（退款/配送/投诉），再转到对应处理流程。这属于哪种模式？

A. Agent（因为 AI 在做分类决策）
B. Workflow — Routing（路由分发）
C. Workflow — Prompt Chaining（提示链）
✅ 正确！虽然 AI 在「分类」这一步做了决策，但「退款问题→走退款流程，配送问题→走配送流程」这个路由规则是预定义的。AI 不能自主决定跳出这些路径去做别的事情，所以是 Workflow - Routing，不是 Agent。记住：判断是 Workflow 还是 Agent，看的是「路径是谁决定的」，不是「谁在做决策」。

🧪 小测验 3-B

你的团队在构建客服 AI 系统，有人提议「我们直接上 Agent，让 AI 自己决定怎么处理所有问题」。根据 Anthropic 的建议，你应该怎么回应？

A. 同意，Agent 比 Workflow 更先进，应该直接用 Agent
B. 不同意，Agent 不适合客服场景
C. 建议先用 Workflow 覆盖常见场景，只在需要灵活性的复杂场景才考虑 Agent
✅ 正确！Anthropic 明确建议「finding the simplest solution possible」—— 能用简单方案就别上复杂方案。客服 80% 的问题是流程固定的（退款、配送、常见投诉），用 Workflow 就够了，而且更可控、更便宜、更好排查问题。只有那 20% 真正需要灵活应变的复杂 case，才值得上 Agent。技术选型不是越先进越好，而是越匹配越好。
