01

Prompt 工程 2020—2023

💡 定义

Prompt 工程 = 通过精心设计输入文本（提示词），引导大模型生成高质量输出。

这是人类与大模型交互的第一种范式。你的全部输入就是 AI 唯一的信息来源——它不知道你有什么文档，不记得昨天聊了什么，也不能去查任何系统。

📱
类比：给陌生人发短信。
收信人很聪明，但只看到这一条短信。回答质量 100% 取决于你写得有多清楚。

📎 权威出处

📄
GPT-3 论文：Brown et al.《Language Models are Few-Shot Learners》
2020年5月 · OpenAI · 31位作者
arxiv.org/abs/2005.14165
首次系统展示 Few-Shot Prompting：在提示词中给几个示例，模型就能学会新任务，无需微调。
📄
Chain-of-Thought 论文：Wei et al.《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》
2022年1月 · Google Research · NeurIPS 2022
arxiv.org/abs/2201.11903
发现在提示词中加入推理步骤，可以显著提升模型的推理能力。

🔑 核心技巧

技巧原理示例
Zero-Shot不给示例，直接问「把这段翻译成英文」
Few-Shot给几个示例定义标准「好的回复像这样：…，差的像这样：…」
Chain-of-Thought要求分步推理「请一步一步思考」
角色设定激活对应领域知识「你是一个有 10 年经验的客服培训师」
格式约束减少猜测输出形式「用 JSON 格式回答，包含 reason 和 score 两个字段」

⚠️ 天花板

  • 无记忆：每次对话全新开始，之前教的全忘
  • 无外部数据：不知道你公司的文档、流程、规则
  • 无行动能力：只能回答文字，不能操作任何系统
  • 上下文窗口有限：早期 GPT-3.5 仅 4K token ≈ 3000 字
🎯 天花板公式：Prompt 上限 = 你的表达能力 × 模型自身知识。当你发现「怎么写提示词都差点意思」的时候，就该进入第二阶段了。

📊 Andrew Ng 的实测数据

📄
Andrew Ng《Agentic Design Patterns Part 1》· 2024年3月20日
deeplearning.ai

Ng 用 HumanEval 编码基准做了对比：

方式准确率说明
GPT-3.5 Zero-Shot（纯 Prompt）48.1%一次生成，不迭代
GPT-4 Zero-Shot（纯 Prompt）67.0%更强模型，但仍是一次生成
GPT-3.5 + Agent 循环95.1%弱模型 + 迭代验证，碾压强模型的一次性输出
💡 关键结论：迭代流程（Agent 循环）带来的提升，远大于换一个更强模型。这就是为什么行业一路从 Prompt 往后面演进。

🧪 小测验 1

你想让 AI 分析一份 100 页的客服手册，用 Prompt 工程能搞定吗？

A. 能，把 100 页全粘贴进去就行
B. 能，分 10 次粘贴，每次 10 页
C. 不太行，需要上下文工程来解决
✅ 正确！100 页远超 token 限制；分批粘贴则丢失跨批上下文。这正是上下文工程（RAG）要解决的问题——让 AI 动态检索所需段落，而不是全塞进去。
02

上下文工程 2023—2025

💡 定义

上下文工程 = 在 AI 回答之前，系统地管理它能「看到」的全部信息。

Prompt 工程关心的是「怎么问」，上下文工程关心的是「拿什么来答」。它不是一种技术，而是一组技术的总称：

🧠
CoT 思维链

教 AI 怎么想

🔍
RAG 检索增强

给 AI 看什么

📝
Memory 记忆

让 AI 记住什么

🔧
Tool 描述

告诉 AI 能用什么

📎 权威出处

📄
RAG 原始论文：Lewis et al.《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》
2020年5月 · Facebook AI / UCL / NYU · NeurIPS 2020
arxiv.org/abs/2005.11401
首次提出 RAG 架构：检索外部知识 + 生成回答，解决模型知识过时和幻觉问题。
🐦
「Context Engineering」术语：Tobi Lütke（Shopify CEO）
2025年6月19日 · X/Twitter
原文：「I really like the term 'context engineering' over prompt engineering. It describes the core skill better.」
Andrej Karpathy 于 6月23日回复「+1 for context engineering」表示认可。
📄
Anthropic 专文：《Effective Context Engineering for AI Agents》
2025年9月29日
anthropic.com
定义：「considering the holistic state available to the LLM at any given time and what potential behaviors that state might yield.」

📖 RAG 如何工作

一句话总结 RAG：不是把 100 页全塞给 AI，而是只给它最相关的几段。就像你不会把整个图书馆搬给朋友，而是帮他找好那几页。

🆚 Prompt 工程 vs 上下文工程

对比维度Prompt 工程上下文工程
关注点怎么写提示词AI 能看到什么信息
知识来源模型训练数据你的文档、数据库、API
能处理的信息量受限于上下文窗口无上限（检索后只取片段）
是否可溯源不能，可能幻觉可以，标注出处
实时性截止到训练日期知识库随时更新
03

Workflow 工作流 2024

💡 定义

Workflow = 通过预定义的代码路径，编排多个 LLM 调用和工具。

关键词是「预定义」——执行路径是人类设计好的，AI 只在每个节点上做决策，不能自主改变流程。

🏭
类比：流水线。
工人（AI）在每个工位上干活，但工位顺序、转运规则都是工厂设计好的。工人不能自己决定跳过某个工位或增加一个新工序。

📎 权威出处

📄
Anthropic《Building Effective Agents》· 2024年12月19日
anthropic.com
原文：「Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents, on the other hand, are systems where LLMs dynamically direct their own processes and tool usage.」
这是行业首次清晰区分 Workflow ≠ Agent。

🧩 Anthropic 定义的五种 Workflow 模式

模式工作方式适用场景
① Prompt Chaining
提示链		A 的输出是 B 的输入，串行执行先翻译 → 再摘要 → 再格式化
② Routing
路由分发		分类输入，分发到不同处理路径客服：退款问题→退款流程；配送问题→物流系统
③ Parallelization
并行处理		同一任务同时拆分给多个 LLM同时翻译成 3 种语言；多角度审核同一份代码
④ Orchestrator-Workers
编排-执行		主 LLM 拆解任务，分配给多个工人 LLM「分析这份报告」→ 主 LLM 拆成 5 个子任务 → 分别执行
⑤ Evaluator-Optimizer
评估-优化		一个 LLM 生成，另一个打分反馈，循环改进写文案 → 评估质量 → 给修改建议 → 重写 → 直到达标

⚠️ 为什么要单独讲 Workflow？

最常见的误区：很多人把 Workflow 和 Agent 混为一谈。Anthropic 在这篇文章中特别强调：

「When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed. This might mean not building agentic systems at all.」

翻译：能用 Workflow 解决的，别上 Agent。Workflow 更可控、更便宜、更可预测。

🧪 小测验 2

用户发消息给客服 AI，系统先分类问题类型（退款/配送/投诉），再转到对应处理流程。这属于哪种模式？

A. Agent（因为 AI 在做决策）
B. Workflow — Routing（路由分发）
C. Workflow — Prompt Chaining（提示链）
✅ 正确！虽然 AI 在「分类」这一步做了决策，但整体流程路径是预定义的（退款→退款流程，配送→物流查询）。AI 不能自主决定跳出这些路径，所以是 Workflow - Routing，不是 Agent。
04

Agent 智能体 2023—2024

💡 定义

Agent = LLM 自主决定执行路径和工具使用，动态控制自己的工作流程。

跟 Workflow 的本质区别：Workflow 的路径是人定义的，Agent 的路径是 AI 自己规划的。

🏨
类比：管家 vs 前台。
Workflow 是前台——你问什么，按手册查答案。Agent 是管家——你说「帮我订明天去北京的行程」，他自己去查航班、比价、订票、订酒店、安排接机。路径由他自己决定。

GitHub 首个全自主 GPT-4 Agent 演示项目。上线一周 Star 数破万，引爆「AI Agent」全球热潮。

🧠 Agent 的三大支柱（Lilian Weng 定义）

🆚 Workflow vs Agent

维度WorkflowAgent
执行路径人类预定义AI 自主规划
可预测性✅ 高（路径固定）⚠️ 低（路径动态）
灵活性❌ 低（不能处理意外）✅ 高（可即兴应对）
成本可控⚠️ 不可控（可能反复试错）
适用场景规则明确的重复任务需要判断力的开放任务
Anthropic 的建议：「Workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed at scale.」
—— 能用 Workflow 就用 Workflow，只有需要灵活性时才上 Agent。
05

Harness 运行外骨骼 2025—2026

💡 定义

Harness = 让 Agent 能够跨越多个上下文窗口、持续数小时甚至数天完成大型任务的基础设施。

Agent 的致命弱点：上下文窗口是有限的。一个复杂项目可能需要几十轮对话、上百次工具调用，早就超出单次会话的容量了。Harness 解决的就是这个问题。

🏗️
类比：轮班制工厂。
想象一个软件项目，每个工程师只工作 4 小时就走人，下一个人来接班，但接班的人对之前的进度一无所知。Harness 就是那套交接班制度——进度文档、任务清单、验收标准——确保每个「班次」都能无缝接上前一班的工作。

📎 权威出处

📄
Anthropic《Effective Harnesses for Long-running Agents》
2025年11月26日
anthropic.com
原文：「The core challenge of long-running agents is that they must work in discrete sessions, and each new session begins with no memory of what came before. Imagine a software project staffed by engineers working in shifts, where each new engineer arrives with no memory of what happened on the previous shift.」
📄
Anthropic《Harness Design for Long-running Application Development》
2026年3月24日
anthropic.com/engineering
续篇：聚焦 Harness 设计模式在前沿 Agentic Coding 中的应用。

🛠️ Anthropic 的 Harness 方案：双 Agent 架构

⚠️ 没有 Harness 会怎样？

Anthropic 在论文中总结了两种典型失败模式：

失败模式表现后果
一口吃成胖子Agent 试图在单个 context window 中完成全部工作上下文耗尽，半成品代码，留下一堆残废 feature
虚假完成新的 context window 启动后，Agent 看到之前的代码就认为「已经做完了」过早宣布完工，实际上大量功能未实现
Anthropic 的洞察：「Inspiration for these practices came from knowing what effective software engineers do every day.」—— Harness 的设计灵感来自人类优秀工程师的日常工作习惯：写进度文档、做增量提交、跑测试验证。

🧪 小测验 3

一个 Agent 在开发大型项目时，每次新会话都从零开始理解代码，导致重复工作。应该用 Harness 的哪个机制解决？

A. 给它更大的上下文窗口
B. claude-progress.txt 持久化进度
C. 换一个更强的模型
✅ 正确！Anthropic 的方案是用 claude-progress.txt + Git 历史作为跨会话的持久记忆。新会话启动后先读进度文件，就知道前面做到哪了。更大窗口和更强模型都不能根本解决「跨会话失忆」问题。
06

Loop 自主循环 2026

💡 定义

Loop = Agent 在 Harness 的支撑下，按不同触发条件和停止条件自主循环执行任务，直到目标达成。

这是前五层能力的集大成：Prompt 定义指令 → 上下文工程提供知识 → Workflow/Agent 执行任务 → Harness 保障跨会话连续性 → Loop 把这一切串成自动化闭环。

🏭
类比：自动化工厂。
前五层是「教一个人怎么干活」。Loop 是把整条生产线自动化——原料进去、产品出来，你只在异常告警时才介入。

🔄 四种 Loop 类型（Claude 官方定义）

核心分类维度：如何触发（Triggered by）× 如何停止（Stop criteria）

① Turn-based Loop 回合制循环

触发方式用户发一条 Prompt
停止条件Claude 判断任务完成 or 需要更多信息
适用场景短期任务：修 Bug、写文档、代码优化
人类角色提需求 → 等结果 → 验收

🔑 每条 Prompt 启动一个自主循环：理解 → 执行 → 验证 → 必要时重复

② Goal-based Loop 目标循环 · /goal

触发方式用户手动发 Prompt（实时）
停止条件目标达成 OR 达到最大轮次上限
关键机制小型快速模型在每轮后评估是否达标
适用场景有量化目标的任务：测试覆盖率 ≥ 80%、Lighthouse ≥ 90 分
人类角色只定义「成功是什么样」，不管「怎么到达」

③ Time-based Loop 定时循环 · /loop /schedule

触发方式时间表 / 定时器
停止条件时间到 or 完成指定迭代次数
适用场景定期任务：每天总结 Slack、每小时检查 CI、每日日报
人类角色设置一次，之后只看结果
/loop 5m check my PR, address review comments, and fix failing CI
// 每 5 分钟检查一次 PR，处理 review 意见，修复失败的 CI

④ Proactive Loop 主动循环 · Dynamic Workflows

触发方式事件驱动（新 PR、新 Issue、代码提交等）
停止条件事件处理完毕
关键机制串联 trigger（触发器）+ goal（目标）+ orchestration（编排），全流程自动
适用场景完全无人值守：自动 Code Review、自动修复、自动部署
人类角色只看最终结果

📊 四种 Loop 的递进关系

Claude 官方的分类逻辑：你每升一层，交出去的东西就多一样。

Loop 类型你交出了什么你还保留什么
① 回合制交出执行保留触发、检查、判断
② 目标循环交出执行 + 检查保留触发、最终判断
③ 定时循环交出执行 + 检查 + 触发保留异常干预
④ 主动循环交出一切只看结果
⚠️ 重点不是循环越多越好。选哪种 Loop 取决于你对 AI 的信任度和场景的容错空间。一个精准的回合制循环，可能比一个失控的全自动循环好 10 倍。

🧪 小测验 4

你希望 AI 每天早上自动生成解决率日报并推送到群里。应该用哪种 Loop？

A. 回合制——每天手动跟 AI 说「帮我看看」
B. 目标循环——设定「解决率 ≥ 85%」为目标
C. 定时循环——/schedule 每天 09:00 执行
D. 主动循环——让 AI 自己决定什么时候生成
✅ 正确！日报是固定时间 + 固定流程的任务，定时循环最合适。目标循环适合「优化到某个值」，主动循环适合「事件驱动」。这里的触发条件是时间，不是事件。
