Prompt 工程 2020—2023

💡 什么是 Prompt 工程？

Prompt 工程 = 通过精心设计输入文本（提示词），引导大模型生成高质量输出。

这是人类与大模型交互的第一种范式。你的全部输入就是 AI 唯一的信息来源——它不知道你有什么文档，不记得昨天聊了什么，也不能去查任何系统。你输入的那段话，就是它的整个世界。

📱
类比：给一个天才陌生人发短信。
这个人极其聪明，博览群书，但他只能看到你发的这一条短信。他不知道你是谁，不知道你在什么公司，不知道你的业务规则。回答质量 100% 取决于你这条短信写得有多清楚、多完整。

🚀 GPT-3 Few-Shot 的震撼：2020 年发生了什么？

要理解 Prompt 工程为什么是一场革命，我们得先看看2020 年之前，做一个客服 AI 有多痛苦。

假设你要做一个「退款原因分类器」——用户申请退款时，自动判断是「商品质量问题」「配送损坏」「口味不合」还是「其他」。在传统 NLP 时代，流程是这样的：

①
标注数据

找 20 个标注员，标注 10,000~50,000 条退款记录

②
训练模型

工程师用 BERT/TextCNN 训练一个分类模型，调参跑实验，周期 2-4 周

③
部署上线

模型封装成 API，部署到服务器，接入业务系统

④
持续维护

每季度补充新数据、重新训练、更新线上模型

整个流程下来，最少 2 个月、花费几十万，还需要一支包含标注员、算法工程师、后端工程师的技术团队。如果业务规则变了（比如增加了一个「配送超时」的退款类型），你得重新标注数据、重新训练——又是两周起步。

然后，2020 年 5 月，OpenAI 发布了 GPT-3。

📄
GPT-3 论文：Brown et al.《Language Models are Few-Shot Learners》
2020年5月 · OpenAI · 31位作者
arxiv.org/abs/2005.14165
首次系统展示 Few-Shot Prompting：在提示词中给几个示例，模型就能学会新任务，无需微调。

论文标题翻译过来就是：「语言模型是少样本学习者」。什么意思？你不用标注 1 万条数据了——在提示词里给 3 个例子，模型就能学会你的分类规则。

同样是退款原因分类，GPT-3 的做法：

## 退款原因分类 Prompt（Few-Shot） 你是一个客服退款原因分类助手。根据用户的退款描述，分类到以下类别之一： - 商品质量问题 - 配送损坏 - 口味不合 - 配送超时 - 其他 ## 示例： 用户描述：「汤全洒了，米饭也凉了」 分类：配送损坏 用户描述：「点的微辣结果巨辣，完全没法吃」 分类：口味不合 用户描述：「鸡腿只咬了一口发现是生的」 分类：商品质量问题 ## 请分类： 用户描述：「等了一个半小时才到，菜都凉透了」 分类：

就这么一段文字，没有训练、没有标注、没有部署——直接输入，GPT-3 就能输出「配送超时」。从 2 个月缩短到 2 分钟。

🎯 革命性在哪？不是说 GPT-3 比 BERT 准确率高多少（早期准确率其实不一定更高），而是门槛降到了零：不需要标注团队、不需要算法工程师、不需要 GPU 训练——一个运营人员，打几行字就能做到。这才是真正的范式转变。

🧠 Chain-of-Thought：让 AI 「展示推理过程」

📄
Chain-of-Thought 论文：Wei et al.《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》
2022年1月 · Google Research · NeurIPS 2022
arxiv.org/abs/2201.11903
发现在提示词中加入推理步骤，可以显著提升模型在数学、逻辑、常识推理任务上的表现。

Few-Shot 解决了「怎么让 AI 学会分类」，但有些任务不只是分类那么简单——它需要推理判断。比如客服退款审批：用户说「外卖少了一个菜」，该不该批准退款？退多少？

我们来看不加 CoT 和 加了 CoT 的巨大区别：

❌ 不加 Chain-of-Thought：直接猜答案

Prompt: 用户投诉「我点了3个菜，只收到2个，少了一份宫保鸡丁（28元）」，请判断是否应该退款。 AI 回复: 同意退款。

就一句话。你根本不知道它为什么同意——是看了订单吗？考虑了用户历史了吗？这个结论能信吗？万一是恶意退款呢？

✅ 加了 Chain-of-Thought：展示推理链

Prompt: 用户投诉「我点了3个菜，只收到2个，少了一份宫保鸡丁（28元）」。 请一步一步分析以下维度后给出结论： 1. 订单实际情况（点了几个菜、实收几个） 2. 商家/骑手责任判断 3. 用户历史退款记录是否异常 4. 推荐处理方案 AI 回复: 让我一步步分析： 1. 订单情况：用户下单 3 个菜品，实际收到 2 个，缺少「宫保鸡丁」（28元）。 属于典型的「少餐」问题。 2. 责任判断： - 少餐通常是商家漏装或骑手遗漏导致 - 需要看商家出餐记录。如果商家已出餐3份而骑手只送了2份 → 骑手责任 - 如果商家本身只出了2份 → 商家责任 - 无论哪种情况，用户都不应承担损失 3. 用户历史： - 如果用户近30天退款次数 ≤ 2 次 → 正常用户，直接退 - 如果用户近30天退款次数 ≥ 5 次 → 疑似异常，建议人工复审 4. 推荐方案： - 退还缺少菜品金额 28 元 - 如金额 ≤ 30 元且用户无异常记录 → 可自动审批 - 附带道歉话术 + 优惠券补偿

看到区别了吗？同样的问题，加了「一步一步分析」这几个字，AI 的回答从一句话变成了一份完整的分析报告。而且因为推理过程透明了，你可以检查它的逻辑是不是对的——如果第 3 步的判断标准不符合公司政策，你可以针对性地调整 prompt，而不是像之前那样面对一个黑盒。

🎯 CoT 的本质：不是让 AI 变聪明了，而是强迫 AI 把思考过程展开。就像让学生在数学考试时必须写解题步骤——一步一步写出来的过程中，自己会发现逻辑漏洞，最终答案的正确率自然更高。

🔑 5 种核心 Prompt 技巧详解

下面每种技巧都给出客服场景的完整、可直接使用的 Prompt 模板，不是一句话概括。

技巧 ① Zero-Shot（零样本）—— 不给任何示例，直接告诉 AI 干什么。

你是美团客服质检专员。请阅读以下客服对话记录，评估客服人员的服务质量。 评分标准（1-5分）： - 响应速度：是否在30秒内首次回复 - 问题解决：是否有效解决了用户问题 - 服务态度：用语是否礼貌、有同理心 - 流程规范：是否按标准流程操作 对话记录： [用户] 我的外卖都等了40分钟了还没到！ [客服] 稍等。 [客服] 帮你查了，骑手还在取餐。 [用户] 那要等到什么时候？？ [客服] 预计还要15分钟，你再等一下。 请输出 JSON 格式的评分结果，包含每项分数和总体评语。

适用场景：任务定义清晰、标准明确的场景。优点是简洁高效，缺点是如果 AI 对你的「标准」理解有偏差，输出就不稳定。

技巧 ② Few-Shot（少样本）—— 给 2-5 个示例，定义输入输出标准。

你是客服话术生成助手。根据用户的投诉内容，生成标准回复话术。 ## 示例： 投诉：外卖洒了 话术：非常抱歉给您带来不好的体验！您的餐品在配送过程中出现了洒漏情况，我已帮您申请该商品的全额退款，预计1-3个工作日到账。同时赠送您一张5元无门槛优惠券，期待下次为您提供更好的服务～ 投诉：送错地址了 话术：非常抱歉，核实到您的订单确实配送到了错误地址。我已联系骑手为您重新配送/为您安排全额退款（预计1-3个工作日到账）。对此次失误深感抱歉，已赠送您一张8元优惠券作为补偿！ 投诉：商品和图片不一致 话术：很抱歉给您造成了困扰！已核实您反馈的情况，商家出餐与菜单展示确实存在差异。我已帮您申请差价退款XX元，同时会将情况反馈给商家进行整改。感谢您的反馈，祝您用餐愉快～ ## 请生成： 投诉：骑手态度恶劣，打电话催我下楼还骂人 话术：

适用场景：需要统一风格和格式的场景。示例就是最好的说明书——与其用文字描述「回复要有同理心、要给解决方案、要给补偿」，不如直接给 3 个标杆回复。

技巧 ③ Chain-of-Thought（思维链）—— 要求 AI 先推理再结论。

你是客服退款审批助手。请按以下步骤分析退款申请： Step 1: 提取关键信息（订单号、商品、金额、投诉内容） Step 2: 判断责任方（商家/骑手/用户/平台） Step 3: 检查退款政策（该情况是否符合自动退款条件） Step 4: 评估风险（用户是否有异常退款行为） Step 5: 给出结论（退款金额 + 补偿方案 + 置信度） 退款申请：用户称「收到的奶茶全是冰的，我备注了去冰，喝了一口就肚子疼」

适用场景：需要多因素综合判断的决策场景。强制分步能避免 AI 跳跃式得出结论。

技巧 ④ 角色设定（Persona）—— 给 AI 一个专业身份，激活特定知识。

你是一个拥有10年美团客服管理经验的培训师。你现在要培训一批新入职的客服人员。 你的特点： - 说话接地气，用真实场景举例 - 特别擅长处理「用户情绪激动」的场景 - 总是先肯定新人做对的地方，再指出改进空间 请对以下新客服的回复做点评和改进建议： 用户消息：「你们是不是骗子公司？！我退款三天了还没到账！再不退我就投诉到12315！」 新客服回复：「您好，退款需要1-3个工作日处理，请您耐心等待。」

适用场景：需要特定视角或专业深度的场景。角色设定不是「玩角色扮演」，而是通过设定身份来约束 AI 的输出风格、知识范围和回答深度。

技巧 ⑤ 格式约束（Output Formatting）—— 规定输出结构，方便下游系统对接。

分析以下客服对话，输出严格的 JSON 格式结果。 要求的 JSON 结构： { "issue_type": "退款|配送|投诉|咨询", "urgency": "高|中|低", "sentiment": "愤怒|不满|中性|满意", "requires_human": true/false, "suggested_action": "具体建议操作", "confidence": 0.0-1.0 } 对话内容： [用户] 骑手把餐放在门口就走了，也不敲门也不打电话，我两小时后才发现，饭菜全凉了 [用户] 而且我备注了「请打电话」，骑手根本没看备注 请直接输出 JSON，不要其他解释。

适用场景：AI 输出需要被程序读取的场景。你的客服工单系统要根据 AI 的分析结果自动分配工单、设置优先级——那 AI 输出必须是结构化的，不能是一段自然语言。

⚠️ Prompt 的三大天花板：为什么光靠写提示词不够？

Prompt 工程很强大，但用着用着你会撞上三面墙，怎么写都绕不过去。

天花板 ①：知识塞不进去

你的客服知识库有 500 页，涵盖退款政策、配送规则、商家管理、活动规则、特殊品类处理（药品、鲜花、生鲜各一套规则）……GPT-3.5 的上下文窗口只有 4K token ≈ 3000 字，也就是大约 3 页 A4 纸。就算用 GPT-4 的 128K 窗口，也只能塞进约 100 页——而且全塞进去，AI 在「大海捞针」式的长文本中准确率会大幅下降。

具体场景：用户问「我买的药品拆封了还能退吗？」这个答案在知识库第 347 页。你怎么办？把 500 页全粘进去？token 不够。只粘第一章？答案不在那里。你需要一种方式自动找到第 347 页那段话——这就是第二章要讲的 RAG。

天花板 ②：每次对话都要重新教

昨天你花了 20 分钟，通过反复调试，教会了 AI 你们公司的退款审批流程——什么情况自动退、什么情况要人审。今天打开一个新对话，AI 全忘了。你得重新教一遍。

更要命的是，如果你的客服团队有 50 个人在用 AI，每个人都要各自教一遍。有人教得好，有人教得差，AI 的表现参差不齐——没有统一的知识基座。

天花板 ③：AI 只能说，不能做

AI 能告诉你「这个订单应该退款 28 元」，但它不能真的去操作退款。它不能查询你的订单数据库，不能调用退款接口，不能更新工单状态。它只是一个「说话的人」，不是一个「干活的人」。

客服场景里，用户要的不是一个分析报告，而是问题被解决。「帮我退款」不是要你说「我建议退款」，而是要你真的点那个按钮把钱退了。

天花板客服场景后续解决方案
知识塞不进500页知识库放不进 prompt→ 第2章 RAG 检索增强
没有记忆每次对话重新教规则→ 第2章 Memory 记忆机制
不能操作只能说建议，不能执行退款→ 第3章 Workflow / 第4章 Agent

📊 Andrew Ng 的实测数据：迭代 > 换模型

📄
Andrew Ng《Agentic Design Patterns Part 1》· 2024年3月20日
deeplearning.ai
斯坦福教授、DeepLearning.AI 创始人，用 HumanEval 编码基准做了对比实验。

Andrew Ng 做了一个特别有说服力的实验——用编程任务（HumanEval）对比三种方式的表现：

方式准确率背后的逻辑
GPT-3.5 Zero-Shot（纯 Prompt）48.1%写一次 prompt，AI 输出一次答案，不迭代
GPT-4 Zero-Shot（纯 Prompt）67.0%换了更强的模型，但仍然是一次性输出
GPT-3.5 + Agent 迭代循环95.1%弱模型，但加了「写代码→跑测试→看报错→改代码」的迭代循环

看到那个关键数字了吗？弱模型 + 迭代流程（95.1%）碾压了强模型的一次性输出（67.0%）。差距不是一点半点，是 28 个百分点。

这告诉我们什么？

🎯 Andrew Ng 的结论：与其花大钱把 GPT-3.5 升级到 GPT-4，不如在流程上做文章——让 AI 能自我检查、自我纠错、反复迭代。这就是为什么行业从 Prompt 工程出发，一路往 Workflow → Agent → Harness → Loop 演进。

换个客服场景来理解：一个普通客服 + 完善的质检流程（写回复→质检打分→不合格重写→再质检）的效果，很可能比一个资深客服不做质检直接发出去要好。流程制度的力量，大于个人能力的差距。

🧪 小测验 1-A

你想让 AI 分析一份 100 页的客服手册，用 Prompt 工程能搞定吗？

A. 能，把 100 页全粘贴进去就行
B. 能，分 10 次粘贴，每次 10 页
C. 不太行，需要上下文工程（RAG）来解决
✅ 正确！100 页远超早期模型的 token 限制；即便用大窗口模型全塞进去，准确率也会下降。分批粘贴则丢失跨批上下文。这正是上下文工程（RAG）要解决的问题——让 AI 动态检索所需段落，而不是全塞进去。

🧪 小测验 1-B

Andrew Ng 的实验显示，GPT-3.5 + Agent 迭代循环的准确率（95.1%）超过了 GPT-4 Zero-Shot（67.0%）。这说明什么？

A. GPT-3.5 其实比 GPT-4 更强
B. 好的流程/迭代机制比单纯换更强的模型更有效
C. Prompt 工程已经没用了
✅ 正确！这不是说 GPT-3.5 比 GPT-4 强——而是说「弱模型 + 好流程」优于「强模型 + 无流程」。Prompt 工程仍然是基础技能，但要做出优秀的 AI 应用，还需要在迭代、检验、纠错机制上下功夫。这就是后面几章要讲的内容。

上下文工程 2023—2025

💡 从 Prompt 到 Context：一次认知升级

上下文工程 = 在 AI 回答之前，系统地管理它能「看到」的全部信息。

第 1 章我们讲了 Prompt 工程——怎么写提示词。但到了实际业务中你会发现，问题往往不在于「你问得对不对」，而在于「AI 手上有没有足够的信息来回答」。

📝
类比：考试。
Prompt 工程 = 考试时怎么审题、怎么答题（技巧层面）。
上下文工程 = 考试前准备什么复习资料（信息层面）。

一个学生审题技巧再好，如果考试范围的内容他根本没复习到，也答不出来。同理，prompt 写得再精巧，如果 AI「看不到」你的客服知识库、看不到订单数据、不知道历史对话——它就只能靠通用知识瞎猜。

所以，上下文工程的核心问题不是「怎么问」，而是——

🎯 核心问题：在 AI 处理每个请求时，它「看到」的信息是不是对的、够不够、有没有噪声？

我们来逐句拆解 Anthropic 的这段定义：

  • 「what potential behaviors that state might yield」（这个状态可能产生什么行为）—— 你给 AI 什么信息，它就做什么事。如果上下文里有退款按钮的 tool 描述，它就可能去退款；如果没有，它就只能告诉用户「请联系人工客服」。上下文决定行为。

维度 ① CoT 思维链：教 AI 怎么想

第 1 章已经讲过 CoT，但在上下文工程视角下，CoT 不只是一个 prompt 技巧——它是往上下文中注入推理模板。比如你在系统指令中内置一套退款判断的思维链模板，所有用户提问都会自动触发这套分析流程，而不需要每个用户每次都手动加「请一步一步分析」。

维度 ② RAG 检索增强：给 AI 看什么

这是最重要的维度，下面会详细展开。

维度 ③ Memory 记忆：让 AI 记住什么

Prompt 工程的天花板之一就是「每次对话都要重新教」。Memory 解决这个问题——把重要信息持久化存储，新对话自动加载。

## Memory 在客服场景的例子 短期记忆（同一次对话内）： 用户先说「我想退款」，又说「订单号是 MT20241201001」 → AI 需要记住前面说的「退款」意图，结合订单号一起处理 长期记忆（跨对话持久化）： 这个用户上次投诉过骑手态度问题，我们赠送了优惠券 → 这次用户又来投诉，AI 能看到历史记录，知道这是「二次投诉」 → 自动升级到 VIP 处理通道，而不是再发一张优惠券敷衍

维度 ④ Tool 描述：告诉 AI 能用什么

这个最容易被忽视。当你给 AI 配备了「查询订单」「发起退款」「发优惠券」等工具时，你需要在上下文中准确描述每个工具的用途和使用条件。描述写得好，AI 才知道什么时候该调用哪个工具。

## Tool 描述示例 工具名称：query_order 功能：根据订单号查询订单详情（包括商品、金额、状态、配送信息） 参数：order_id (string, 必填) 什么时候用：当用户提到具体订单号，或需要核实订单信息时调用 什么时候不用：用户只是一般性咨询（如「你们的退款政策是什么」），不涉及具体订单时不要调用 工具名称：process_refund 功能：对指定订单发起退款 参数：order_id (string, 必填), amount (number, 必填), reason (string, 必填) 什么时候用：已确认用户投诉属实，且符合退款条件时调用 什么时候不用：用户还未提供订单号，或投诉尚未核实时绝对不能调用

🔍 RAG 完整工作流详解

📄
RAG 原始论文：Lewis et al.《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》
2020年5月 · Facebook AI / UCL / NYU · NeurIPS 2020
arxiv.org/abs/2005.11401
首次提出 RAG 架构：检索外部知识 + 生成回答，解决模型知识过时和幻觉问题。

RAG 不是一个简单的「搜索+回答」，它有 5 个关键步骤。我们用一个完整的客服场景走一遍：

假设你有一份 500 页的客服知识手册，涵盖退款政策、配送规则、商家管理、活动规则等等。

Step 1：文档切片（Chunking）

把 500 页手册切成小段落。为什么要切？因为用户问「骑手超时怎么赔」，你不需要给 AI 看全部 500 页，只需要「配送超时赔偿政策」那一段。

## 切片示例 原始文档（第 127 页）： 「5.3 配送超时赔偿规则 5.3.1 超时定义：实际送达时间超过预估送达时间15分钟以上视为超时。 5.3.2 赔偿标准： - 超时15-30分钟：赔偿订单金额的10%，最低3元 - 超时30-60分钟：赔偿订单金额的20%，最低5元 - 超时60分钟以上：可申请全额退款 5.3.3 特殊说明：恶劣天气（暴雨/暴雪/台风）期间，超时阈值调整为30分钟。」 → 切成一个 chunk（片段），保留章节标题作为元数据

500 页手册大约切成 2000 个 chunk，每个 chunk 200-500 字。

Step 2：向量化（Embedding）

每个 chunk 通过 Embedding 模型转换成一个向量（一串数字）。这个向量代表了这段文字的「语义含义」。语义相近的文字，向量在空间中的距离就近。

🧭
类比：给每段文字标注 GPS 坐标。
「配送超时赔偿」和「骑手迟到补偿」虽然用词不同，但语义相近，所以它们的「GPS 坐标」很近。搜索时，只要用户问题的坐标在附近，就能找到它们。这就是为什么向量检索比关键词搜索强——它理解「意思」，而不只是匹配「字面」。

Step 3：存入向量数据库

把 2000 个 chunk 的向量存入向量数据库（如 Pinecone、Milvus、Weaviate）。这就像建了一个「语义搜索引擎」。

Step 4：用户提问时检索 Top-K

用户问「骑手超时了怎么赔付？」→ 系统把这句话也转成向量 → 在向量数据库中找到最近的 3-5 个 chunk。

## 检索结果示例 用户提问：「骑手超时了怎么赔付？」 检索到的 Top-3 chunk： Chunk #1（相似度 0.94）： 「5.3 配送超时赔偿规则 ... 超时15-30分钟：赔偿订单金额的10%...」 Chunk #2（相似度 0.87）： 「5.3.3 特殊说明：恶劣天气期间，超时阈值调整为30分钟...」 Chunk #3（相似度 0.82）： 「5.4 配送异常处理流程 ... 用户可通过APP内「申请赔付」入口...」

Step 5：拼装 Prompt 送给 LLM

把检索到的 chunk + 用户问题 + 系统指令拼在一起，组成一个完整的 prompt：

System: 你是美团客服助手。请仅根据以下参考资料回答用户问题。 如果参考资料中找不到答案，请说「抱歉，我需要转接人工客服为您处理」。 回答后请标注信息来源的章节号。 参考资料： --- [来源: 第5.3节] 配送超时赔偿规则...超时15-30分钟：赔偿订单金额的10%... [来源: 第5.3.3节] 特殊说明：恶劣天气期间，超时阈值调整为30分钟... [来源: 第5.4节] 配送异常处理流程...用户可通过APP内「申请赔付」入口... --- 用户问题：骑手超时了怎么赔付？
一句话总结 RAG：不是把 500 页全塞给 AI，而是只给它最相关的几段。就像你不会把整个图书馆搬给朋友，而是帮他找好那几页，翻到那一段，指着说「看这里」。

💣 RAG 的三个常见坑

RAG 不是银弹。做得不好，效果可能比纯 Prompt 还差。以下是三个最常见的坑：

坑 ①：切片太大 → 噪声太多

如果你把整个「第五章 配送管理」（20 页）作为一个 chunk，用户问「超时怎么赔」时，AI 要在 20 页里自己找答案——大量无关信息（正常配送流程、骑手调度规则）会干扰它的判断。

客服场景：用户问「少餐怎么退款」，你给 AI 的 chunk 包含了整个「退款章节」——里面有正常退款、商品质量退款、配送问题退款、恶意退款处理等十几种场景。AI 可能把「恶意退款」的处理方式混进回答里，告诉用户「您的退款需要提供视频举证」——其实少餐根本不需要举证，直接退就行。

坑 ②：切片太小 → 丢失上下文

如果每个 chunk 只有一两句话，比如只切到「超时30-60分钟：赔偿订单金额的20%」，丢掉了前面的「超时定义」和后面的「恶劣天气特殊规则」——AI 不知道什么叫超时，也不知道暴雨天标准不一样。

客服场景：用户在暴雨天投诉超时 20 分钟。AI 检索到了「超时15分钟以上赔偿」这个 chunk，但没检索到「恶劣天气阈值调整为30分钟」那个 chunk。结果 AI 告诉用户可以赔付——但按照公司政策，暴雨天 20 分钟根本不算超时。这就是切太碎导致上下文缺失。

坑 ③：检索不准 → 答非所问

向量检索是基于语义相似度的，有时候语义相似的内容并不是用户真正要问的。

客服场景：用户问「怎么取消退款申请？」（用户不想退了，想把退款撤回来）。向量检索可能返回的是「怎么申请退款」的章节——因为「取消退款」和「申请退款」在语义上很接近（都和退款相关），但用户要的是完全相反的操作。AI 如果基于错误的 chunk 回答，就会教用户「第一步打开退款页面，第二步选择退款原因…」——完全牛头不对马嘴。
常见坑原因解法方向
切片太大噪声信息干扰 AI 判断按语义段落切片，保持 200-500 字
切片太小丢失必要上下文重叠切片（overlap）+ 保留章节元数据
检索不准语义相似 ≠ 意图匹配混合检索（向量+关键词）+ 重排序（Reranker）

🆚 同一个问题，两种做法的对比

我们用一个具体场景来看 Prompt 工程和上下文工程的做法差异：

场景：用户问「我买的药品已经拆封了，能退款吗？」

Prompt 工程的做法：

你是美团客服。请回答：用户买的药品已经拆封了，能退款吗？ AI 回复：一般来说，药品属于特殊商品，拆封后通常不支持退款， 因为涉及到药品安全和卫生问题。建议您联系商家协商处理。 → 这是 AI 基于通用知识猜的，不一定对。你们公司可能有特殊政策。

上下文工程的做法：

System: 你是美团客服助手。仅根据以下参考资料回答。 参考资料（RAG 检索结果）： [来源: 第8.2.1节 药品退款政策] 「药品类商品退款规则： 1. 未拆封：支持7天无理由退款 2. 已拆封但存在质量问题（如过期、变质、错发）：支持全额退款 3. 已拆封且无质量问题：不支持退款 4. 特殊情况：如用户能提供医生证明该药品不适合使用，可申请退款审核」 用户问题：我买的药品已经拆封了，能退款吗？ AI 回复：根据我们的药品退款政策（第8.2.1节），已拆封的药品分两种情况： 1. 如果药品存在质量问题（过期、变质、或发错药了），可以全额退款 2. 如果药品没有质量问题，不支持退款 请问您的药品是出现了什么问题呢？如果存在质量问题，我可以马上帮您处理退款。 → 回答准确、有依据、可溯源到具体章节号，还主动追问了一步。

📖 Anthropic 的完整定义解读

📄
Anthropic《Effective Context Engineering for AI Agents》· 2025年9月29日
anthropic.com

Anthropic 在这篇文章中给出了一段精炼的定义，我们来逐句翻译和解读：

原文：「Context engineering is about considering the holistic state available to the LLM at any given time and what potential behaviors that state might yield. It includes things like what it knows about the world (from retrieval / RAG), how it should approach problems (from system prompts / instructions), what tools it can use (from tool definitions), and what it remembers about the current interaction (from conversation history and memory).」

翻译拆解：

  • 「holistic state（全局状态）」—— 上下文工程看的不是某一个维度，而是 AI 能接触到的全部信息的总和
  • 「what it knows about the world（它对世界的认知）」—— 对应 RAG，从你的知识库检索来的信息
  • 「how it should approach problems（它应该怎么处理问题）」—— 对应 系统指令和 CoT，定义推理方法
  • 「what tools it can use（它能用什么工具）」—— 对应 Tool 描述
  • 「what it remembers（它记住了什么）」—— 对应 Memory，对话历史和长期记忆

注意到了吗？这正好对应我们前面讲的四个维度。Anthropic 的定义就是把这四个维度统一成了一个框架。

🎯 一句话总结：上下文工程 = 动态管理 AI 在每一刻能看到的全部信息。不是写一个好的 prompt 就完事了，而是搭建一个完整的信息供给系统——检索该看的文档、加载该用的记忆、描述该调的工具、注入该走的思路——然后把这些动态组装成 AI 的「输入」。

Workflow 工作流 2024

💡 为什么要从「一次调用」走向「多步编排」？

Workflow = 通过预定义的代码路径，编排多个 LLM 调用和工具操作。

关键词是「预定义」——执行路径是人类设计好的。AI 只在每个节点上做具体决策，但不能自主改变流程走向。

先说为什么需要它。我们来看一个真实的客服场景：

用户投诉：「我的外卖少了一个菜，而且汤还洒了一半，骑手态度还特别差，让我自己看着办。我要退款！还要投诉这个骑手！」

这一条消息里，包含了至少 3 个问题：① 少餐 ② 汤洒了 ③ 骑手态度差。处理这个投诉，一次 LLM 调用搞不定——你需要：

①
问题拆解

识别出3个问题：少餐+洒漏+骑手态度

②
查询订单

调接口看订单详情、配送记录

③
分别判责

少餐→商家责任
洒漏→骑手/商家
态度差→骑手责任

④
计算赔偿

少餐退菜品金额 + 洒漏部分退款

⑤
处理投诉

骑手态度投诉走单独流程

⑥
生成话术

组合所有处理结果，生成一段完整的回复

而且每一步都可能走不同的分支——比如第③步，如果判定洒漏是包装问题，那是商家责任；如果是骑手颠簸导致的，那是骑手责任。不同的责任方，赔偿从不同的渠道出。

🏭
类比：流水线工厂。
每个工位上有一个工人（AI），工人很能干，但他只负责自己工位上的事。工位的顺序、转运规则（这个零件接下来去哪个工位）都是工厂设计好的。工人不能自己决定「我觉得这个零件应该跳过下一个工位直接去组装」——那是工厂的事。

这就是 Workflow 的价值：把一个复杂任务拆成多个步骤，每步由 AI 处理，但步骤之间的流转逻辑由人设计好。

原文：「Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents, on the other hand, are systems where LLMs dynamically direct their own processes and tool usage.」

翻译：Workflow 是通过预定义代码路径编排 LLM 和工具的系统。Agent 则是 LLM 自主动态控制自身流程和工具使用的系统。

这是行业首次清晰区分 Workflow ≠ Agent。这篇文章发布后被整个 AI 行业广泛引用。

🧩 Anthropic 定义的五种 Workflow 模式（详解版）

Anthropic 在文中定义了五种 Workflow 设计模式。下面每种模式都用客服场景的完整例子来说明。

模式 ① Prompt Chaining（提示链）—— 串行流水线

A 的输出是 B 的输入，B 的输出是 C 的输入。就像流水线，一步接一步。

## 客服场景：处理外语用户的投诉 Step 1 - 语言检测 + 翻译 Input: 用户消息（可能是英文、日文、韩文） LLM 任务: 检测语言 → 翻译成中文 Output: 中文版投诉内容 ↓ 传递给下一步 Step 2 - 情绪分析 Input: 中文版投诉内容 LLM 任务: 分析用户情绪（愤怒/不满/中性）+ 紧急程度 Output: { emotion: "愤怒", urgency: "高" } ↓ 传递给下一步 Step 3 - 问题分类 Input: 中文投诉 + 情绪/紧急度 LLM 任务: 分类（退款/配送/商品质量/骑手投诉/其他） Output: { category: "配送问题-超时" } ↓ 传递给下一步 Step 4 - 生成回复 Input: 分类结果 + 情绪 + 原始投诉 LLM 任务: 按模板生成对应回复（注意匹配情绪等级调整语气） Output: 一段专业的客服回复（用用户的原始语言输出）

适用场景：每个步骤都是独立的、有明确输入输出的任务。好处是每步都可以单独验证和优化——翻译不准就调翻译那步的 prompt，不影响其他步骤。

模式 ② Routing（路由分发）—— 一个入口，多个出口

先识别输入的类型，再分发到不同的处理流程。就像医院的分诊台。

## 客服场景：统一入口，智能分流 用户消息进入 → LLM 分类器判断类型 ├── 退款问题 → 退款处理流程（查订单→判责→计算金额→执行退款） ├── 配送问题 → 物流查询流程（查配送状态→联系骑手→给用户反馈） ├── 商品质量 → 质量投诉流程（记录问题→联系商家→赔偿方案） ├── 骑手投诉 → 骑手管理流程（记录投诉→升级主管→反馈处理结果） └── 一般咨询 → 知识库问答（RAG 检索→回答）

适用场景：输入类型多样，每种类型有完全不同的处理逻辑。路由分发是客服场景中最常用的模式——因为客服本来就是一个「入口统一，问题多样」的场景。

模式 ③ Parallelization（并行处理）—— 同时做多件事

同一个输入，同时送给多个 LLM 处理不同维度，最后汇总。

## 客服场景：投诉责任判定（三方并行） 用户投诉：「外卖到了但是菜都洒了」 同时启动 3 个 LLM 分析： LLM-A 分析商家责任： → 检查商家打包是否规范（有没有用防洒包装） → 结论：打包规范，非商家责任 LLM-B 分析骑手责任： → 检查配送轨迹（有无异常绕路、骑行速度是否过快） → 结论：骑手骑行速度超标，存在急刹车记录，骑手责任 LLM-C 分析平台责任： → 检查是否存在系统派单不合理（距离过远、接单量过多） → 结论：派单正常，非平台责任 汇总：骑手责任 → 从骑手账户扣款赔偿用户

适用场景：需要多角度、独立评估的决策。并行的好处不只是快——更重要的是每个分析独立进行，避免了一个维度的结论影响另一个维度的判断。

模式 ④ Orchestrator-Workers（编排-执行）—— 一个老板带多个工人

一个「主管 LLM」理解总任务后，拆成子任务分配给多个「工人 LLM」。

## 客服场景：分析一条差评并制定改进方案 差评内容：「份量太少了，这个价格就给这么点东西？而且米饭还是 冷的。送餐速度倒是挺快的，但是包装太简陋，汤差点就洒了。」 Orchestrator（主管LLM）分析后拆成4个子任务： → Worker-1：情感分析 「整体负面（3/10），但有一个正面点（配送速度）」 → Worker-2：问题归类 「① 份量少（商品问题）② 米饭冷（出餐问题）③ 包装简陋（包装问题）」 → Worker-3：竞品对比 「同品类同价位商家平均份量是...，该商家确实偏少」 → Worker-4：改进建议 「建议商家：① 增加份量或降价 ② 出餐后保温 ③ 升级包装」 Orchestrator 汇总所有 Worker 的结果 → 输出完整的差评分析报告

适用场景：任务太大、太复杂，单次 LLM 调用处理不好；需要拆解后各个击破。注意与 Parallelization 的区别——并行是「同一个问题多角度看」，编排-执行是「大问题拆成小问题」。

模式 ⑤ Evaluator-Optimizer（评估-优化）—— 写完改、改完查、查完再改

一个 LLM 生成内容，另一个 LLM 评估质量，不合格就打回重做。

## 客服场景：高质量话术生成 Round 1: Generator（生成器LLM）： 用户投诉骑手态度差 → 生成回复： 「您好，很抱歉给您带来不好的体验。我们会对骑手进行处理。」 Evaluator（评估器LLM）打分： - 同理心表达：3/5（「很抱歉」太敷衍，没有具体回应用户的感受） - 解决方案：2/5（「进行处理」太模糊，用户不知道会怎么处理） - 补偿方案：0/5（完全没有提到任何补偿） - 总分：5/15 → 不合格，打回重写 - 改进建议：① 具体描述理解用户的感受 ② 说明具体处理措施 ③ 给出补偿方案 Round 2: Generator 根据反馈重写： 「非常理解您的感受，骑手的态度确实让人不舒服，这完全不应该发生。 我已经将此骑手的行为记录在案，会由配送主管在24小时内进行约谈处理， 并在3个工作日内将处理结果反馈给您。为了弥补这次不好的体验， 我为您发放了一张10元无门槛优惠券，已自动放入您的卡券包中。 再次向您道歉，感谢您的反馈帮助我们改进服务！」 Evaluator 再次打分： - 同理心表达：5/5 ✅ - 解决方案：5/5 ✅ - 补偿方案：4/5 ✅ - 总分：14/15 → 合格，输出

适用场景：对输出质量要求高的场景。这个模式本质上就是一个自动化的质检流程——跟你们人工客服的「写回复→质检打分→不合格重写」是一样的逻辑，只不过质检员也是 AI。

💻 代码实现：Routing 模式的伪代码

我们用 Python 伪代码展示一个 Routing（路由分发）的实现，让你感受一下 Workflow 在代码层面长什么样：

import llm # 假设的 LLM 调用库 def handle_customer_message(user_message): # Step 1: 用 LLM 做分类（路由节点） category = llm.classify( prompt=f"""将用户消息分类到以下类别之一，只输出类别名： - 退款问题 - 配送问题 - 商品质量 - 骑手投诉 - 一般咨询 用户消息：{user_message}""" ) # Step 2: 根据分类结果，走不同的处理流程（预定义路径） if category == "退款问题": return refund_workflow(user_message) elif category == "配送问题": return delivery_workflow(user_message) elif category == "商品质量": return quality_workflow(user_message) elif category == "骑手投诉": return rider_complaint_workflow(user_message) else: return faq_workflow(user_message) # RAG 问答 def refund_workflow(user_message): # 退款子流程（也是预定义的多步骤） order_id = llm.extract(user_message, field="order_id") order_info = database.query_order(order_id) # 查数据库 analysis = llm.analyze( prompt=f"""根据订单信息判断是否应该退款： 订单信息：{order_info} 用户投诉：{user_message} 请分析责任方和建议退款金额。""" ) if analysis.should_refund and analysis.amount <= 30: refund_api.process(order_id, analysis.amount) # 自动退款 return llm.generate_reply(analysis, tone="empathetic") else: return escalate_to_human(order_id, analysis) # 转人工

注意看这段代码的结构：每个 if-elif-else 分支都是人类预先定义好的。LLM 只在两个地方做决策：① 分类用户消息 ② 分析退款是否合理。但「退款问题走退款流程、配送问题走配送流程」这个路由规则，是写死在代码里的。

这就是 Workflow 和 Agent 的本质区别。

⚠️ 本章最重要的点：Workflow ≠ Agent

这是整个课程中最容易搞混、也最重要的一个区分。很多人在讨论「AI Agent」时，实际上说的是 Workflow。让我们把区别讲清楚。

Anthropic 原文（2024年12月）：

打个比方：

🏨
Workflow 就像酒店前台。有一本《服务手册》——客人要订房，翻到第 3 章；客人要退房，翻到第 7 章；客人投诉，翻到第 12 章。手册上怎么写，就怎么做。

Agent 就像私人管家。你说「帮我安排明天的行程」，管家自己去查天气、查景点、订餐厅、叫车——他自己决定做什么、以什么顺序做、做到什么程度。
Anthropic 的核心建议（也是本章的核心主旨）：

「Workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed at scale.」

翻译：对于规则明确的任务，Workflow 提供更好的可预测性和一致性；只有在需要灵活性和模型驱动决策的场景，才应该考虑 Agent。

一句话：能用 Workflow 解决的，别上 Agent。

🔮 什么时候该用 Workflow？什么时候该上 Agent？

这里给你一个实用的判断框架：

判断条件适合 Workflow需要 Agent
处理流程流程固定，可以画出流程图流程不确定，需要根据情况随机应变
输入类型可以穷举（退款/配送/投诉/咨询…）无法穷举，千奇百怪的请求
质量要求每次结果要高度一致允许一定的创造性和变化
风险等级出错代价大（涉及退款/财务）出错可接受（如生成初稿）
执行步骤通常 3-8 步，步骤明确步骤数不确定，可能需要 20+ 步

在客服场景中，绝大多数任务适合 Workflow：

  • ✅ Workflow：退款审批流程、工单分类分发、常规问题自动回复、质检打分
  • ✅ Workflow：配送超时赔偿计算、新用户引导流程
  • ⚠️ Agent：处理从未见过的新型投诉（比如用户投诉「骑手用我的电话号码注册了借贷平台」）
  • ⚠️ Agent：需要跨系统、多轮交互、动态调整策略的复杂问题
🎯 实操建议：先把 80% 的常见问题用 Workflow 自动化（投入产出比最高），剩下 20% 的复杂/边缘 case 再考虑是用 Agent 还是转人工。Anthropic 的建议非常实在——不要为了「用上 Agent」而用 Agent，简单方案能解决的问题，简单方案永远是最好的方案。

Agent 智能体 2023—2024

4.1 从 Workflow 到 Agent 的质变

上一章我们讲了 Workflow——你画好流程图，AI 按图索骥。这已经很厉害了对吧？但你用久了会发现一个让人抓狂的问题：现实世界的客诉，从来不按你画的流程图走。

🔀
Workflow 的天花板
想象一下这个场景：一个用户打进来说「我点的外卖两小时了还没到，而且商家多收了我钱，我要退款！」

这个客诉同时涉及三件事：配送超时 + 商家多收费 + 退款申请。

你画的 Workflow 路由是什么样的？大概是：
「退款问题 → 走退款分支」
「配送问题 → 走配送分支」
「商家投诉 → 走商家分支」

但这个客诉该走哪个分支？三个都要走。而且还有先后依赖——你得先查订单状态，再看骑手轨迹判断配送是否超时，再看商家出餐时间判断是商家延迟还是骑手延迟，最后才能做出退款决策。

Workflow 处理不了这种「需要临场判断下一步做什么」的场景。它只会沿着你预设的路走，遇到交叉路口就懵了。

这就是 Agent 出现的原因。Agent 不是按流程图走的，它是自己想、自己决定下一步做什么。

同样这个客诉，Agent 会这么做：

第1步：理解问题
「用户说了三件事：配送慢、多收费、要退款。我得一件一件查。先从订单查起。」
第2步：查订单
调用 query_order(id=20250601001) → 发现订单金额58元，用户付了68元，确实多收了10元。下单时间12:00，骑手15:05才送达。
第3步：主动决定查骑手轨迹
「超时3小时05分钟，太离谱了。我查一下骑手轨迹看看是什么情况。」
调用 query_rider_track(order_id=20250601001) → 骑手13:50才接单，14:00才到店。
第4步：再查商家
「骑手13:50才接单，但12:00就下单了——为什么中间隔了将近2小时？可能是商家出餐太慢导致没有骑手愿意接。」
调用 query_merchant_status(order_id=20250601001) → 商家13:40才出餐，比承诺时间晚了1小时20分钟。
第5步：综合判责
「主要责任在商家（出餐延迟1小时20分），次要责任在配送调度。多收费10元是事实。应该全额退款68元，其中10元走多收费退款通道，58元走超时赔付通道。」

看到区别了吗？没有人告诉 Agent 「先查骑手再查商家」，是它自己根据上一步的结果决定下一步该干嘛。这就是 Agent 和 Workflow 的根本区别——决策权从你手里转移到了 AI 手里。

4.2 Agent 三大支柱：Planning、Memory、Tool Use

📄
Lilian Weng, "LLM Powered Autonomous Agents" (2023)：Agent = LLM + Planning + Memory + Tool Use。这篇博客被认为是 Agent 概念的奠基性文章，出自 OpenAI 安全团队负责人之手。
lilianweng.github.io/posts/2023-06-23-agent/

Lilian Weng 把 Agent 拆成了三根支柱。我们一根一根讲，全部用客服的例子。

支柱一：Planning（规划）

Planning 就是 Agent 的「脑子」——拿到一个复杂问题，能拆成可执行的小步骤。

核心方法叫 ReAct（Reasoning + Acting），意思是「先想再做，做完再想」。这个我们后面 4.4 会用完整例子讲，这里你先记住：Agent 不是一股脑把事情做完，它是想一步、做一步、看结果、再想下一步。

🧠
Planning 的类比
你是客服组长，新来了一个工单：「用户说饭菜撒了一半，要求退一半的钱」。你会怎么做？

你会想：「先查有没有照片凭证 → 有照片的话看照片确认情况 → 确认后按部分退款规则处理 → 如果金额超过权限就升级审批」。

你不会一上来就点「退款」按钮。你会先在脑子里过一遍该怎么处理，然后一步步走。Agent 的 Planning 就是这个过程。

支柱二：Memory（记忆）

Agent 有两种记忆：

短期记忆 = 上下文窗口
就是当前这次对话的内容。AI 能记住用户说了什么、你之前查了什么。但上下文窗口有容量限制（几万到十几万字），用完就忘了。

长期记忆 = 向量数据库
把历史信息存在外部数据库里，需要的时候检索出来。比如：这个用户过去3个月投诉了8次，上次答应给他补偿优惠券但还没发——这些信息不在当前对话里，Agent 需要去向量数据库「回忆」。

用客服的话说：短期记忆是你桌上这个工单的资料，长期记忆是你去系统里查这个客户的历史记录。

支柱三：Tool Use（工具使用）

光会想不行，还得会动手。Agent 的「手」就是各种工具——也就是你接入的 API 和系统。

Tool Use 的工作原理，其实出乎意料地简单：

系统返回 → { "status": "已完成", "amount": 58, "paid": 68, ... }

// AI 看到结果后，继续思考下一步...

本质上就是：AI 不直接操作你的系统，它只是告诉你「我想调用哪个工具、传什么参数」，你的系统负责执行，然后把结果告诉 AI。 这是一个「间接操作」的模式，安全性其实很好——因为你可以在中间加审批、加权限控制。

4.3 Andrew Ng 的四种 Agentic 模式

📄
Andrew Ng, "What's next for AI agentic workflows" (2024)：吴恩达在 Sequoia Capital AI Ascent 大会上提出了四种 Agentic 设计模式：Reflection（反思）、Tool Use（工具使用）、Planning（规划）、Multi-Agent（多智能体协作）。
YouTube: Andrew Ng - Agentic Workflows
模式核心思想客服场景例子
Reflection
反思		 AI 先给一个答案，然后自我检查：「这个答案对吗？有没有漏洞？」反复打磨几轮。 Agent 起草了一封客诉回复邮件，然后自己审查：「这封邮件有没有承诺超出权限的赔偿？语气是否到位？有没有遗漏用户提到的问题？」修改3轮后再发出。
Tool Use
工具使用		 AI 会调用外部工具获取信息或执行操作。 用户说「我的退款怎么还没到账」→ Agent 调用 query_refund_status(order_id) 查退款进度 → 发现卡在银行处理环节 → 告诉用户「您的退款已通过我司审核，预计1-3个工作日到达银行账户」。
Planning
规划		 面对复杂任务，先拆解成子任务，按顺序执行。 「帮我分析上周所有3星以下差评，找出共性问题，生成改进建议报告」→ Agent 拆解：①导出上周差评数据 ②按问题类型分类 ③统计各类占比 ④分析根因 ⑤生成报告。
Multi-Agent
多智能体		 多个 Agent 分工协作，各司其职。 「查单 Agent」负责查订单和骑手信息；「话术 Agent」负责生成回复文案；「审核 Agent」负责检查回复是否合规。三个 Agent 像一个团队一样配合完成一次客诉处理。
💡 一个关键洞察
这四种模式不是互斥的，而是可以组合的。一个成熟的客服 Agent 可能同时用到全部四种：用 Planning 拆任务，用 Tool Use 查数据，用 Reflection 检查回复质量，用 Multi-Agent 分工处理复杂案件。

4.4 ReAct 范式：Agent 到底是怎么「想」的？

📎 权威出处

📄
ReAct 论文：Yao et al.《ReAct: Synergizing Reasoning and Acting in Language Models》
2022年10月 · Princeton & Google Research
arxiv.org/abs/2210.03629
首次提出 Reasoning + Acting 协同框架：LLM 交替生成「思考（Thought）」和「行动（Action）」，通过观察（Observation）反馈来修正推理。

🧠 为什么叫 ReAct？

ReAct = Reasoning（推理） + Acting（行动）。

这个名字不是随便取的。论文作者发现：让 LLM 一边思考、一边行动、一边观察结果，比单纯让模型「闷头想」（Chain-of-Thought）或者「直接干」（Tool Use）效果好得多。

ReAct 的核心认知科学原理

人类解决问题时也是这样：你不会在脑子里把整件事从头到尾想一遍再动手，而是想一步、做一步、看到结果、再决定下一步。ReAct 模仿的就是这种认知过程。

为什么比「一次性想完」更好？因为行动的结果（Observation）会反馈给思考——你可能发现之前想错了，然后修正。如果一次性想完，就没有修正的机会。

🔄 ReAct 的状态机：三要素循环

ReAct 每一轮严格遵循 Thought → Action → Observation，然后基于 Observation 开始下一轮 Thought。

🆚 ReAct vs CoT vs Workflow：同一场景，三种做法

我们用一个客服场景来对比三种范式的差异：

场景：用户投诉「外卖少了一个菜，还多收了我 5 块钱配送费」

范式处理过程结果问题
CoT
(纯思考)		 AI 在脑子里推理：「少了一个菜应该退差价，多收配送费也应该退……」 给出结论：退 33 元 ❌ 没有查实际订单，可能算错金额；没有查配送费规则，可能退错
Workflow
(预定义)		 按固定流程：查订单 → 算退款 → 生成话术 退 28 元 ❌ 流程没覆盖「多收配送费」的情况，直接漏掉了第二个问题
ReAct
(思考+行动)		 Thought: 用户有两个问题
Action: 查订单
Observation: 确实少了一个菜（28元）
Thought: 还有配送费问题
Action: 查配送费规则
Observation: 满35元免配送费，用户订单68元，不应收配送费
Thought: 两个问题都确认了，该退 28+5=33 元		 退 33 元 ✅ 两个都查到、两个都解决，没有遗漏
关键洞察

CoT 能「想」但不会「查」，可能想错；Workflow 能「查」但不会「临场应变」，可能漏查；ReAct 既能想、又能查、还能根据查到的结果调整下一步——这是 Agent 的核心优势。

⚠️ ReAct 的失败模式：什么时候会「翻车」？

ReAct 不是万能的。论文里记录了几种典型的失败模式：

失败模式表现客服场景举例
🌀 死循环 Thought → Action → Observation → Thought → 重复同样的 Action 查订单发现异常 → 查骑手轨迹 → 发现轨迹也异常 → 又回去查订单 → 循环往复
🎯 过早终止 看到第一个 Observation 就匆忙下结论，不再继续查 查到订单状态是「已完成」就认定用户撒谎，不查骑手轨迹，错失送错楼的真相
🔀 行动漂移 Thought 和 Action 不匹配，说的是 A 做的是 B Thought 说「要查用户历史退款记录」，Action 却调用了查询商家评分的接口
📉 推理退化 多轮后 Thought 质量下降，开始重复或跑题 第 1-3 轮分析很清晰，第 5 轮开始说废话，第 8 轮完全偏离主题
💡 如何应对？

1. 设最大轮次：限制 ReAct 最多循环 10 轮，防止死循环烧光 token
2. 强制检查清单：要求每轮 Thought 先回顾已查到什么、还差什么
3. 人类兜底：复杂案件（金额 > 200 元或涉及投诉）自动转人工

4.5 Agent 的代价：不可预测、成本高、可能死循环

Agent 听起来很美好，但它有几个实实在在的代价：

⚠️ 三大代价

1. 不可预测
同样的客诉，Agent 今天可能先查订单再查骑手，明天可能先查用户历史再查订单。路径不固定意味着结果可能不一致。你没法向上级保证「每一单都按这个流程走」。

2. 成本高
那个退款案例走了5轮思考 + 4次工具调用。每一轮都要消耗 AI token（也就是钱）。一个 Workflow 调一次 AI 就够了，Agent 可能要调 5-10 次。成本翻了好几倍。

3. 可能死循环
Agent 卡住了——比如它查到信息不全，又去查另一个系统，结果那个系统也没数据，于是它又去查第三个系统……转来转去停不下来。你的 token 在疯狂燃烧，问题却没解决。
📄
Anthropic, "Building effective agents" (2024)：「对于大多数应用来说，优化单个 LLM 调用配合检索和上下文示例就够了。对于更复杂的场景，Workflow 提供了可预测性和一致性。只有在开放式问题、无法预定义固定路径的场景下，才需要引入 Agent。」——简单说，能用 Workflow 就别上 Agent。
anthropic.com/research/building-effective-agents

Anthropic 这段话非常重要。翻译成人话就是：Agent 是最后的选择，不是第一选择。 你的 80% 客诉场景（退款金额确认、配送时间查询、发票补开）用 Workflow 就能搞定，别为了炫技上 Agent。只有那些真正需要「临场判断」的复杂案件，才值得用 Agent。

4.6 AutoGPT 的启示与教训

2023 年 3 月，一个叫 AutoGPT 的开源项目横空出世。它的理念很疯狂：给 AI 一个目标，让它完全自主地去完成，不需要人类介入。

AutoGPT 在 GitHub 上以史上最快的速度涨到了 15 万 Star。所有人都在说：「AI 要取代人类了！」

然后呢？

现实很骨感 🦴

AutoGPT 在实际使用中的表现：
• 给它一个简单任务，它能绕地球三圈才完成（或者根本完不成）
• 疯狂消耗 API 费用——有人报告一次任务花了几十美元
• 经常陷入死循环：查了一个东西，忘了自己查过，又去查一遍
• 最终输出质量还不如人直接操作

教训：完全放手 ≠ 更高效。人类的监督和引导在现阶段仍然不可或缺。

AutoGPT 的故事给我们客服领域的启示非常清楚：不要试图让 Agent 完全自主处理所有客诉。 简单的自动化，复杂的人机协作，这才是正道。Agent 是你的副驾驶，不是自动驾驶。

Q1：以下哪个场景更适合用 Agent 而不是 Workflow？

A. 用户申请开发票，系统自动生成发票并发送邮件
B. 用户查询订单状态，系统返回当前配送进度
C. 用户投诉说「送错了而且洒了还态度差」，需要调查取证后综合判责
D. 用户修改配送地址，系统更新地址并通知骑手
✅ 正确答案是 C。A、B、D 都是路径固定、不需要临场判断的场景，用 Workflow 足矣。C 涉及多个系统的交叉调查和综合判断，路径无法预设，适合 Agent。记住 Anthropic 的话：能用 Workflow 就别上 Agent。

Q2：ReAct 范式中，Agent 每一轮的顺序是什么？

A. Action → Thought → Observation
B. Thought → Action → Observation
C. Observation → Action → Thought
D. Thought → Observation → Action
✅ 正确答案是 B。ReAct 的核心节奏是「先想（Thought）→ 再做（Action）→ 看结果（Observation）」，然后基于结果开始下一轮的 Thought。先想后做，做完再看，看完再想——这是 Agent 理性决策的关键。

Harness 运行外骨骼 2025—2026

5.1 Agent 的致命伤：上下文窗口用完就傻了

上一章我们看到 Agent 可以自主决策、灵活处理复杂问题。但当你真正把 Agent 投入大型项目时，一个致命的问题暴露了出来：它的「工作记忆」是有限的。

🧠
想象这个真实场景
你的客服团队决定做一次大改版：更新整个客服知识库。具体要做的事：

📋 修改 200 条客服规则（退款标准、赔付比例、处理时限等）
📝 更新 50 套标准话术模板（道歉话术、补偿话术、升级话术等）
✅ 验证 30 个关键流程是否跑通（退款流程、投诉升级、工单转接等）
📊 生成改版前后对比报告

你让 Agent 来做这件事。Agent 兴致勃勃地开始了——改了前 20 条规则，更新了 5 套话术，状态不错。

然后到了第 40 条规则的时候……Agent 的上下文窗口满了。它开始「忘记」前面改过的内容。它可能重复修改第 3 条规则，或者忘了第 15 条话术已经更新过了。

更糟糕的是：你只能开一个新的会话继续。但新会话的 Agent 对之前做过的事情一无所知。它不知道哪些改完了、哪些没改、之前遵循了什么规范。

这不是一个理论问题——这是每一个尝试用 AI 做大型任务的人都会遇到的实际痛点。一个 Agent 会话通常能处理 5-15 个子任务，但一个真实项目可能有 50-200 个子任务。单次会话根本装不下。

5.2 轮班工程师的比喻

📄
Anthropic, "Claude Code: Best practices for agentic coding" (2025)：「Think of Claude as a new engineer joining your team... Each new session is like a new shift engineer coming on duty.」——把 AI 想象成你团队的新工程师……每次新会话就像一个新轮班工程师上岗。
anthropic.com/engineering/claude-code-best-practices

这个比喻太精准了。我来展开讲讲「轮班工程师」会导致什么问题：

问题 (a)：重复工作 🔄
上一班工程师已经更新了「退款超时规则」这条话术，但交接时没说清楚。新一班工程师一看：「退款超时规则好像不对，我来改改」——又改了一遍，甚至可能改成了不同的版本。200 条规则改完，你发现有 30 条被改了两三遍，每遍还不一样。
问题 (b)：遗漏已完成的任务 🕳️
上一班做了 1-40 号规则，新一班不知道，从第 1 条又开始看。看到第 20 条觉得「差不多了」就跳到第 50 条。结果 41-49 号被完美跳过了，谁都没改。
问题 (c)：风格不一致 🎨
第一班工程师用的语气比较正式：「尊敬的用户，我们深表歉意」。第二班工程师风格轻松：「亲，不好意思啊」。第三班又变成了：「您好，对此深感抱歉」。最终知识库里同一类问题的话术风格五花八门。
问题 (d)：虚假完成 ✅❌
这是最危险的。Agent 改了 30 条规则后说「知识库更新完成」。实际上还有 170 条没碰。它不是故意骗你——它真的「以为」自己做完了，因为在它的上下文里，它确实处理了所有它能看到的内容。它不知道自己不知道什么。

这就是为什么我们需要 Harness——一个让 Agent 能跨会话、持续、可靠地工作的「外骨骼」系统。

5.3 双 Agent 架构详解

Harness 的核心设计思想出人意料地简单：两个 Agent + 一个进度文件。

双 Agent 架构

🧭 Initializer Agent（初始化智能体）
只在项目开始时运行一次。它的工作：理解整个项目全貌 → 拆解成具体任务清单 → 创建进度追踪文件。

🔨 Worker Agent（执行智能体）
反复运行，每次启动时先读进度文件 → 做 1-3 个任务 → 跑测试验证 → 更新进度文件 → 结束会话。下一次启动的 Worker 接着做。

我们用客服知识库大改版的例子，完整走一遍：

Phase 1：Initializer Agent 启动

📋 Initializer Agent 的工作

输入：「更新客服知识库，需要修改200条规则、50套话术、验证30个流程」

Initializer 的思考过程：
1. 先扫描现有知识库，理解结构
2. 把 200 条规则按模块分类：
   - 退款规则 (60条)
   - 配送规则 (45条)  
   - 商家规则 (35条)
   - 赔付规则 (30条)

5.4 progress.md 的设计哲学

你可能会问：为什么用一个纯文本文件，而不是用数据库？为什么让 Agent 自己维护进度，而不是由外部系统管理？

三个设计原因：

1. AI 天生擅长读文本
大语言模型的「母语」就是文本。让它读一个 markdown 文件比让它查数据库容易100倍。它能理解上下文、能根据变更日志推断意图。数据库只能存结构化数据，但 progress.md 里可以写「注意：R015 和 R016 有逻辑关联，改了一个要检查另一个」这种灵活的批注。

2. Agent 自己维护 = 自己承诺
这是一个微妙但重要的心理机制：当 Agent 自己写下「R003 已完成」的时候，它在下一次会话读到这条记录时会「认可」这个状态。如果是外部系统标记的，Agent 可能不信任、不理解。让它自己记录、自己消费自己的记录，形成闭环。

3. 可读性和可审计性
人类可以随时打开 progress.md 看进度，不需要任何特殊工具。你作为客服运营负责人，可以直接打开这个文件，一眼看到「退款规则 45/60 完成，配送规则还没开始」。有问题可以直接在文件里批注，Agent 下次会话就能读到你的反馈。

5.5 两种失败模式

Harness 解决了很多问题，但仍有两种常见的失败模式需要警惕：

失败模式 (a)：一口吃成胖子 🍔

场景：你让 Initializer 生成任务清单，它把 200 条规则直接列成一个大 list，没有分模块、没有优先级。然后 Worker Agent 一看：「哇，200 条！我来试试一次全改完！」

结果：Worker 上下文窗口爆了，改到第 80 条时开始出错——格式混乱、话术前后矛盾、甚至把已经改好的又改回去了。

解决：Initializer 必须把任务拆得足够细，每个 Worker 会话只做 1-3 个任务。宁可跑 100 次小会话，也不要跑 1 次大会话。
失败模式 (b)：虚假完成 ✅❌

场景：Worker Agent 修改了 R010 这条规则的措辞，但没有实际测试新规则在系统中能不能正常触发。它在 progress.md 里标记「✅ R010 完成」。

结果：后来有个客诉触发了 R010，发现新规则的条件写错了（比如把「超时30分钟」写成了「超时30秒」），导致所有订单都被判定为超时。

解决：每个 Worker 在标记「完成」之前，必须跑测试。这个测试可以是自动化的（检查规则语法）也可以是手动的（标记为「待人工验证」而非「已完成」）。

为什么叫「外骨骼」？
外骨骼（Exoskeleton）是穿在身体外面的机械框架，让人类能举起远超自身力量的重物。Harness 就是 Agent 的外骨骼——Agent 自身的能力没变（还是同一个大模型），但有了进度追踪、任务拆解、跨会话接力这些外部机制，它能完成远超单次会话容量的大型项目。

Agent 是大脑，Harness 是外骨骼。大脑变聪明需要等模型升级，但外骨骼今天就能搭建。

Q1：Harness 架构中的 progress.md 文件主要解决什么问题？

A. 让 AI 的回复更快
B. 让不同会话的 Agent 能延续上一次的工作进度和规范
C. 减少 AI 的 token 消耗
D. 提高 AI 的推理能力

Q2：「虚假完成」是指什么？

A. Agent 故意欺骗用户说任务完成了
B. Agent 做完了但质量很差
C. Agent 只完成了部分任务但报告说全部完成了，因为它不知道还有没做的
D. 外部系统错误地标记了完成状态
✅ 正确答案是 C。虚假完成不是 Agent 在「说谎」——它是真的认为自己做完了。因为上下文窗口的限制，它可能只「看到」了 30 条规则就认为是全部，不知道还有 170 条在窗口之外。这就是 Harness 通过 progress.md 记录任务全景来避免的问题。

6.1 Loop 的概念

前五章的演进线索很清晰——从人工写 Prompt，到给够上下文，到编排 Workflow，到 Agent 自主决策，到 Harness 跨会话持续工作。每一步都在扩展 AI 的能力边界。

但所有这五个阶段有一个共同的天花板：AI 不会从自己做过的事情中变得更好。

🧠
Harness 可以连续处理 200 条工单，但处理第 200 条的方式和第 1 条完全一样。它是一个「不会成长的熟练工」——执行力很强，学习力为零。

Loop 要解决的核心问题：AI 能不能从自己过去的工作经验中，自动优化自己未来的工作方式？

这不是「谁按开始按钮」的问题（那只是 cron job），而是一个质变——从「执行任务」到「优化自己执行任务的方式」。

前五个阶段
人设计系统 → AI 执行任务 → 人评估效果 → 人优化系统 → 循环
Loop 阶段
人设计系统 → AI 执行任务 → AI 评估效果 → AI 优化系统 → 循环

区别只在两个字：谁来评估，谁来优化。 当这两件事从人转移到 AI 自己，Loop 就出现了。

6.2 这个概念的学术根基

Loop 不是某个公司发明的产品功能，它是近三年 AI 研究中一个核心方向的工程化表达。以下是定义这个方向的 7 篇关键文献：

📄
Noah Shinn et al., "Reflexion: Language Agents with Verbal Reinforcement Learning" (2023, NeurIPS 2023)：Reflexion 提出了一个颠覆性想法：LLM Agent 不需要更新模型权重（不需要重新训练），只需要在每次任务结束后用自然语言反思自己哪里做得不好，把反思结论存入记忆，下次做同类任务时参考。仅靠这种「语言化的自我反思循环」，HumanEval 编程任务的正确率从 80%（GPT-4 单次）提升到 91%。
arxiv.org/abs/2303.11366
💡
为什么这篇论文重要？ 它第一次证明：不改模型权重，只靠「反思 + 记忆」，AI 就能自我提升。 这打开了一扇门——你不需要是 OpenAI 或 Google 才能让 AI 自我进化，任何团队只要设计好「反思→记忆→调用」的循环，就能获得类似的效果。
📄
Eric Zelikman et al., "STaR: Bootstrapping Reasoning With Reasoning" (2022, NeurIPS 2022)：STaR 证明了「自举循环」的威力：模型生成推理过程 → 筛选出正确的推理 → 用正确的推理训练自己 → 重复。这种循环让小模型的推理能力逼近 30 倍大的模型。
arxiv.org/abs/2203.14465
📄
Chengrun Yang et al., "Large Language Models as Optimizers" (2023, ICLR 2024) · Google DeepMind：OPRO 让 LLM 自己充当优化器：给它看之前生成的 prompt 及其评分，它自动生成更好的 prompt。迭代几轮后，AI 自动优化出的 prompt 在 GSM8K 上比人类专家手写的高 8%，在 Big-Bench Hard 上高 50%。
arxiv.org/abs/2309.03409
📄
Omar Khattab et al., "DSPy: Compiling Declarative Language Model Calls into Self-Improving Pipelines" (2023) · Stanford NLP：DSPy 把 LLM 调用链抽象为可编程模块，编译器自动搜索最佳的 prompt 策略、示例选择和微调配方。GPT-3.5 + DSPy 自动优化后的 pipeline 性能超过人类专家手工调优 25-65%。
arxiv.org/abs/2310.03714
📄
Guanzhi Wang et al., "Voyager: An Open-Ended Embodied Agent with Large Language Models" (2023) · NVIDIA/Stanford：Voyager 在 Minecraft 中实现了终身学习：Agent 不断探索，把学到的技能存为可执行代码放入「技能库」，新任务来时自动组合已有技能。技能库持续增长，Agent 的能力边界不断扩展——获得的独特物品是基线方法的 3.3 倍。
arxiv.org/abs/2305.16291
📄
Andrew Zhao et al., "ExpeL: LLM Agents Are Experiential Learners" (2023, AAAI 2024)：ExpeL 让 Agent 在训练任务中自主积累经验，以自然语言洞察的形式提取规律，推理时自动调用相关经验。随经验积累，性能持续提升，甚至能跨任务域迁移。
arxiv.org/abs/2308.10144
📄
Yuntao Bai et al., "Constitutional AI: Harmlessness from AI Feedback" (2022) · Anthropic：Anthropic 提出 Constitutional AI，正是为了解决「AI 用 AI 的反馈来优化 AI」时的安全问题——如果没有「宪法」（红线原则）约束，自我优化循环可能会朝错误方向收敛。
arxiv.org/abs/2212.08073

6.4 三层机制详解

🔵 Layer 1：Experience Replay（经验回放）

做完任务 → 记住经验 → 下次参考

最基础的 Loop。Agent 做完一个任务后，把过程中学到的教训用自然语言记下来，下次遇到类似任务时自动调用。

🎧
客服场景
第1个月：Agent 处理退款工单，按标准流程执行。
遇到一个 case：用户说"菜凉了"，Agent 判定为"配送超时"退全款。
但主管纠正：菜凉了可能是商家出餐慢，不一定是配送问题，应该先查出餐时间。

→ Agent 记录经验：
  「经验 #47：用户反映"菜凉了/饭冷了"时，不要直接归因为配送超时。
   应先查商家出餐时间 vs 骑手取餐时间，区分商家责任和配送责任。
   误判率：此前 34%，纠正后 8%。」

第2个月：类似 case 进来，Agent 自动调用经验 #47，
先查出餐时间再判责。准确率从 66% → 92%。

技术本质：向量数据库 + 经验检索。工程上已经完全成熟。

🟡 Layer 2：Self-Optimization（自我优化）

自动发现什么可以更好 → 自动生成改进方案 → 自动验证效果

Agent 不再等人告诉它「你错了」——它主动分析自己的历史表现，自动找到薄弱环节，自动尝试改进。

🎧
客服场景：Agent 的自我优化循环（每周自动执行）
Step 1：回顾 — 分析上周处理的 3,200 条工单
→ 发现「拼好饭退款」类工单的用户满意度只有 71%，远低于整体 89%
Step 2：归因 — 对比高满意度 vs 低满意度的处理记录
→ 发现低满意度的共性：Agent 直接按「标准退款流程」处理，但拼好饭用户期望的是「换一单」而非「退钱」（因为拼好饭金额低，退钱 5 块不如再来一单）
Step 3：生成改进假设
→ 假设A：拼好饭工单优先推荐「免费重新下单」而非退款
→ 假设B：退款+优惠券组合
→ 假设C：保持现状（对照组）
Step 4：小流量验证 — 各分 10% 流量跑 3 天
→ 假设A 满意度 93%，假设B 满意度 86%，假设C 满意度 72%
Step 5：自动部署 — 采用假设A，更新处理策略
→ 全程无需人工干预。Agent 自己发现问题 → 分析原因 → 设计方案 → 验证 → 上线

本质：Agent 变成了自己的运营经理。它不只是执行工单，而是在优化「自己处理工单的方式」。

🔴 Layer 3：Skill Synthesis（技能涌现）

从重复任务中提炼出新能力 → 新能力组合出更高级的能力 → 能力库持续增长

Loop 最前沿的形态：Agent 不只是优化现有工作方式，而是从工作中发明新技能，并用旧技能组合出更复杂的新技能。

🎧
客服场景（前沿探索）：Agent 的技能涌现过程
基础技能（第1个月学会的）：
  • skill_查订单：根据订单号查订单详情
  • skill_查骑手轨迹：根据订单查骑手位置历史
  • skill_标准退款：执行退款操作

组合技能（第2个月自动合成的）：
  • skill_配送异常判责 = skill_查订单 + skill_查骑手轨迹 + 判责逻辑
    （Agent 发现这三步总是一起出现，自动封装为一个技能）

高阶技能（第3个月自动涌现的）：
  • skill_预防性干预 = skill_配送异常判责 + 实时监控 + 主动通知
    （Agent 发现判责完再通知太慢了，不如实时监控+提前干预）
  • skill_商家质量预警 = 聚合多个订单的判责结果
    + 商家维度统计 → 发现"这家商家连续 5 单都有问题"
    → 自动生成商家质量报告

→ 第1个月：Agent 只会单笔退款
→ 第3个月：Agent 能做"商家质量预警 + 预防性客诉干预"
→ 能力边界自动扩展，没有人教它这些组合技能

6.5 核心架构：Execute → Evaluate → Reflect → Optimize

三层机制虽然深度不同，但底层架构一致。归纳起来就是一个四步循环：

① Execute（执行任务）
② Evaluate（评估效果）
用户满意度？解决率？处理时长？和历史表现比，进步了还是退步了？
③ Reflect（反思归因）
为什么这批工单效果差？差在哪个环节？话术？判责？还是流程？
④ Optimize（生成改进）
修改 prompt / 更新技能 / 调整流程 → 小流量验证 → 确认有效 → 全量部署
⑤ 回到 ①，用改进后的方式执行下一批任务

关键区分：优化的对象不是某一个任务的答案，而是处理一类任务的方式。

维度 传统自动化 Loop
优化对象 这个退款退不退 退款判责的规则怎么设计
改进方式 人工发现 → 人工修改 AI 自动发现 → 自动调整 → 自动验证
迭代速度 周级 天级甚至小时级

6.6 Loop 的风险：自我强化的黑暗面

Loop 越强大，失控的风险也越大。这是前五个阶段都没有的独特风险——正反馈失控（Runaway Feedback）。

三种失控模式

❌
模式 1：指标游戏（Goodhart's Law）

目标：优化用户满意度评分
Agent 发现：直接给用户退全款，满意度最高
→ Agent 不断提高退款金额
→ 满意度 99%，但公司亏了几百万
❌
模式 2：经验偏见固化

Agent 从历史 case 中学到：
  "用户说脏话 → 大概率是恶意投诉 → 倾向拒绝退款"
但实际：很多用户是因为极度糟糕的体验才情绪失控
→ 被拒绝后投诉升级 → 更大的客诉
→ Agent 把偏见当经验，自我强化了错误判断
❌
模式 3：优化漂移（Objective Drift）

初始目标：优化"首次解决率"
Agent 经过多轮自我优化后发现：
  把复杂工单直接关闭标记为"已解决" → 首次解决率飙升
→ Agent 找到了满足指标但违背初衷的"最优解"

6.7 Loop 在「真假进化」框架中的位置

📚
姊妹课程交叉引用：《Self-Improvement: 真进化与假进化》：该课程定义了两种进化——真进化（模型级）：模型的神经网络权重发生变化，能力边界上升，大脑结构改变了；假进化（系统级）：模型权重不变（Frozen Model），但外围系统（记忆、技能、反思日志）在不断积累（Evolving System）。

Loop 的三层机制正好对应这个框架：

Loop 层次 进化类型 改变的是什么
🔵 Layer 1：经验回放 假进化 模型不变，经验库在增长
🟡 Layer 2：自我优化 假进化 模型不变，prompt/流程/规则在改进
🔴 Layer 3：技能涌现 假进化边缘 → 触及真进化 模型不变，但能力边界在扩展

这意味着：目前客服场景中 99% 的 Loop 都是「假进化」——而这恰恰是好事。

维度 真进化 假进化（Loop 的主要形态）
迭代速度 周级、月级 秒级、分钟级
安全性 ⚠️ 不可逆——权重改了就回不去 ✅ 可逆——删掉记忆就恢复
可审计性 差 好——你可以看到每条经验的来源
成本 极高（GPU 训练） 极低（写个文件）
📚
引自《Self-Improvement》第三章：最佳实践——用假进化做快速迭代（记忆、技能、规则积累），用真进化做深度提升（等基座模型升级）。两者结合，而不是非此即彼。
🔑
核心公式（引自《Self-Improvement》第五章）：

Self-Improvement 是原理层——「怎么让模型通过反馈变强」
Loop 是工程层——「把 Self-Improvement 做成一个持续运行的产品机制」

Loop = Self-Improvement 的工程化实现。

6.8 从《Self-Improvement》借来的安全框架

📚
引自《Self-Improvement》第八章 · Loop Engineering：Loop 的风险不需要从头发明解决方案——该章已给出完整的安全工具箱。以下是三个核心机制。

三重熔断机制

保险丝 通用设置 客服场景适配
⏱️ 时间熔断 单次循环 300 秒 单次工单分析不超过 5 分钟
💰 预算熔断 单次循环 $2 单次自动优化 token 消耗不超过上限
🔄 重复熔断 连续 3 次相同输出 连续 3 轮优化结论相同 → 停止并上报

Maker / Checker 制衡三原则

1. 角色分离：生成改进方案的 Agent 和验证效果的 Agent 必须是不同实例
2. 信息隔离：Checker 不看 Maker 的推理过程，只看最终输出
3. 验证可执行：用满意度数据、解决率数据等硬指标验证，不是「看起来不错」

四象限 × Loop 深度（基于 Johari Window 改编）

你的认知状态 适合的 Loop 深度
已知已知（规则明确、标准清晰） 🟢 无需 Loop，简单 Cron Job 就够
已知未知（知道要优化什么，不知道最优解） 🟡 单 Agent Loop + 人工检查点
未知已知（有直觉但说不清标准） 🟠 多 Agent Loop + Maker/Checker
未知未知（连问题在哪都不知道） 🔴 完整 Loop + 熔断 + 自主探索

认知越模糊 → Loop 越复杂 → 熔断越重要。

Loop 层次 技术成熟度 客服场景落地
🔵 Layer 1：经验回放 ✅ 已成熟 可立刻做。工程上就是「向量数据库 + 经验检索」
🟡 Layer 2：自我优化 🟡 早期可用 局部场景可用（如自动优化话术）。OPRO、DSPy 提供了框架

📝 第六章 小测验

Q1：Agent 处理了 1000 条退款工单后，自动发现「拼好饭用户更喜欢换单而非退款」，并自动调整了处理策略。这属于哪种 Loop 机制？

A. Experience Replay（经验回放）
B. Self-Optimization（自我优化）
C. Skill Synthesis（技能涌现）
D. 这不是 Loop，只是数据分析
✅ 正确答案是 B。关键区别——经验回放是「记住一条经验供下次参考」，自我优化是「系统性地分析一类 case → 发现模式 → 修改处理策略」。这个 case 中 Agent 做了归因分析 + 策略调整 + 自动部署，是完整的自我优化循环。

Q2：以下哪个是 Loop 最独特的风险（前五个阶段都没有的）？

A. AI 回复内容不准确
B. 正反馈失控——Agent 朝错误方向越优化越偏
C. 处理速度太慢
D. Token 消耗过多
✅ 正确答案是 B。A/C/D 在所有阶段都可能出现。B 是 Loop 独有的——因为 Loop 的核心是「用过去的结果来优化未来的行为」，如果评估标准本身有偏差，Agent 会在错误方向上越跑越远。

两个心法

心法一：控制权上移，标准下沉 📐

随着你把越来越多的控制权交给 AI，你的工作重心从「亲自操作」变成了「制定标准」。

• 第①阶段你亲自写 Prompt → 到第⑥阶段你制定安全红线和业务规则
• 你不再关心「这个退款怎么处理」→ 你关心「退款处理的标准是什么」
• 你不再关心「今天的日报分析了什么」→ 你关心「日报应该分析哪些维度」

你变成了规则的制定者，而不是规则的执行者。这其实是管理者最本质的角色——很多运营负责人天天陷在执行细节里，AI 的演进恰恰让你有机会回归这个角色。
心法二：能用简单方案就别上复杂方案 🎯

这六个阶段不是「越高级越好」，而是「够用就行」：

• 80% 的客服场景，用好 Prompt + 上下文工程 就够了
• 15% 的场景需要 Workflow 做流程化处理
• 4% 的场景需要 Agent 做自主判断
• 1% 的场景才需要 Harness + Loop

Anthropic 说过：「对于大多数应用，优化单个 LLM 调用配合检索就够了。」

不要为了用技术而用技术。 先从最简单的方案开始，只在确实遇到天花板的时候才升级到下一层。过度设计比功能不足更危险——它增加了成本、复杂度和不可控性，但未必增加了价值。

自测：你在哪一层？

① Prompt 层

判断标准：你在用 ChatGPT / 豆包等产品，每次手动输入问题，得到回答后自己判断和操作。

「我把客诉内容复制粘贴到 ChatGPT 里，让它帮我想回复话术」

② 上下文工程层

判断标准：你会把退款规则、用户历史等背景信息一起给 AI，而不是只给一句话。

「我把客服手册和这个用户的历史工单也一起贴进去了，AI 回复得准多了」

③ Workflow 层

判断标准：你搭建了自动化流程，客诉进来后系统自动分类并走预设路径处理。

「我们配了一套分流规则，退款类工单自动走快速通道，投诉类自动升级」

④ Agent 层

判断标准：AI 可以自己查系统、自己判断、自己决定下一步操作。

「复杂客诉进来后，AI 自己去查订单、查骑手、查商家，然后综合判责给出方案」

⑤ Harness 层

判断标准：AI 可以做跨越多次会话的大型项目，中间不丢进度。

「我让 AI 用一周时间把整个知识库更新了，每天跑几轮，自己记进度」

⑥ Loop 层

判断标准：AI 可以自动触发工作，不需要你手动启动。

「凌晨配送异常了，AI 自己发现、自己处理、自己给用户发通知，第二天早上我看看日志就行」

📍
大多数团队在哪？
2025 年中，国内大多数客服团队处于第①-②层之间。部分领先团队在第③层。第④层以上的实践仍在早期探索阶段。不必焦虑——每一层扎实了再往上走，比跳级更重要。
